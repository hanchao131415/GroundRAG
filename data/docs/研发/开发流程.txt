示例科技有限公司 研发开发流程

（参考 Scrum Guide 2020 和 Spotify 工程文化，结合公司实际制定）

一、流程概述

公司采用 Scrum 敏捷开发框架，每个 Sprint 周期为两周（10 个工作日）。Sprint 起止时间为周三至次周周二。研发全流程覆盖需求评审、技术方案设计、编码实现、测试验证、灰度发布、全量上线和回顾复盘七个阶段。各阶段均须在项目管理工具（Jira）中记录产出物和状态。

角色定义：产品经理（PM）负责需求定义和优先级排序；技术负责人（Tech Lead）负责架构决策和技术方案评审；Scrum Master（由轮值工程师担任）负责流程推进和 impediment 清除；团队成员（Engineers）负责方案设计、编码、测试和上线。

二、需求评审阶段

PM 在 Sprint 开始前一周的周四前提交 PRD（Product Requirement Document），PRD 包含：用户故事（User Story）、验收标准（Acceptance Criteria）、优先级（P0/P1/P2）、期望上线时间和业务价值说明。Tech Lead 在周五前完成技术可行性评估，给出工作量和风险评估意见。

需求评审会于每周三下午 14:00-16:00 召开，PM、Tech Lead、相关开发工程师和 QA 工程师参加。评审通过的需求进入 Sprint Backlog，PM 按优先级排序。未通过评审的需求退回 PM，附带明确的修改意见，修改后于下轮评审重新提审，修改周期不超过 3 个工作日。

三、技术方案设计

涉及以下任一情况的需求须编写技术方案：跨多个模块或服务、数据库 Schema 变更、外部系统对接或 API 变更、引入新的技术组件或中间件、对现有架构有重大影响。技术方案（RFC）须包含：背景与目标、系统架构图（C4 模型 Context/Container 级）、数据流和时序图、接口定义（API Spec）、数据库变更脚本（migration）、性能评估（预期 QPS、延迟、资源消耗）、风险与回滚方案。

方案评审由 Tech Lead 发起，邀请相关模块 Owner 和资深工程师参加。评审通过后方案归档至 Confluence 技术方案空间。重大架构变更需技术委员会（Tech Committee）评审通过。

四、编码阶段

开发工程师根据 Sprint Backlog 认领任务，从 main 分支创建 feature 分支进行开发。每日 10:00 召开 15 分钟站会（Daily Standup），同步进度、计划和阻塞项。开发期间每日至少 Push 一次代码至远程仓库（WIP 可标记 Draft PR），防止本地代码丢失。

编码须遵守公司代码规范和测试要求。开发期间如需调整技术方案，须在 PR 描述中说明变更并与 Tech Lead 确认。开发工程师负责编写单元测试和必要的集成测试，测试代码与实现代码在同一 PR 中提交。

五、测试阶段

QA 工程师在测试环境（Staging）执行测试。测试类型包括：功能测试（验证验收标准）、回归测试（确保已有功能不被破坏）、边界测试（极端输入和边界条件）、性能测试（当需求涉及性能指标时）和安全测试（涉及用户数据和权限时）。

缺陷管理：测试发现的缺陷登记至 Jira，按严重程度分级——P0（阻断上线，24 小时内修复）、P1（严重影响用户体验，当前 Sprint 内修复）、P2（一般问题，可延至下个 Sprint）、P3（建议优化，纳入 Backlog 后续排期）。P0 和 P1 缺陷修复后须重新回归测试。测试完成后 QA 出具测试报告，确认功能符合验收标准。

六、上线流程

上线前先在灰度环境（Canary，生产环境 5% 流量）部署并持续观察 30 分钟。监控面板（Grafana）重点观察：HTTP 错误率（5xx 占比 < 0.1%）、API 响应时间（P99 < 500ms）、CPU 使用率（< 80%）、内存使用率（< 85%）、数据库连接数和慢查询。灰度期间发现异常指标时立即执行回滚（kubectl rollout undo），并通知相关人员排查根因。灰度确认正常后分步放量：5% → 25% → 100%，每步间隔 10 分钟。

全量上线后，开发工程师和值班运维工程师持续监控 2 小时。确认无异常后，上线流程结束。上线由 DevOps 平台自动化执行，开发工程师触发并监控。

七、回顾与复盘

每个 Sprint 结束时的周二下午召开 Sprint Retrospective，全员参加。使用 Start/Stop/Continue 框架回顾 Sprint 中的亮点（Continue）、需改进项（Stop）和新思路（Start），产出不超过 3 条行动项（Action Items）并指派责任人，纳入下个 Sprint 追踪执行。

线上故障（P0/P1 级事故）在故障恢复后 24 小时内进行根因分析（RCA，Root Cause Analysis），产出一页纸的事故报告（Postmortem），包含：事故时间线、直接原因和根本原因、影响范围和持续时间、修复措施和预防改进计划、责任人。Postmortem 归档至 Confluence，在事故后第一次全体研发会议上通报。事故复盘从"改进流程"的角度出发，不追责个人。

八、持续改进

每季度末进行一次研发效能回顾，统计指标包括：Sprint 完成率（计划点数 / 实际完成点数）、Lead Time（从需求确认到上线的时间）、Code Review 周期（从 PR 创建到合并的时间）、线上故障次数和 MTTR（平均恢复时间）、测试覆盖率趋势。根据数据制定下季度改进目标和行动计划。效能数据在研发季度会议上公布并讨论。

九、紧急修复流程

线上 P0 级故障（核心功能不可用或数据安全事件）启动紧急修复流程。值班工程师 15 分钟内响应告警并初步定位。确认问题后在 hotfix 分支上修复，修复后走简化 Code Review 流程（至少一位 Reviewer Approve，可电话/企微催促）。修复上线后持续监控 1 小时。故障恢复后按第七章要求完成 RCA 和 Postmortem。紧急修复不计入 Sprint 容量。
