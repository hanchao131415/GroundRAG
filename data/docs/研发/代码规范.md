# 示例科技有限公司 研发代码规范

## 第一章 总则

本规范旨在统一团队编码风格，提高代码可读性、可维护性和可测试性，降低因编码风格不一致引发的 Code Review 摩擦成本。规范参考业界最佳实践（Google Style Guide、PEP 8、Airbnb JavaScript Style Guide）并结合公司实际情况制定。

**适用范围**：公司研发部全体工程师（前端、后端、全栈、算法工程师），以及提交代码至公司代码仓库的外部合作开发者。

**基本原则**：可读性优先于简洁性；一致性优先于个人偏好；规范是团队共识，不是教条，异议通过 RFC（Request for Comments）流程讨论修订。

## 第二章 版本控制规范

所有代码托管在公司 GitLab（gitlab.internal.example.com），使用 Git 进行版本管理。主分支 `main` 始终保持可部署状态，严格禁止直接向 main 推送代码。功能开发在 `feature/<简述>` 分支上进行；Bug 修复在 `fix/<简述>` 分支上；紧急线上修复在 `hotfix/<简述>` 分支上，修复后须同时在 main 和当前 release 分支上合并。

提交信息格式：`<type>(<scope>): <简短描述>`。Type 类型包括：feat（新功能）、fix（修复）、refactor（重构，不改变行为）、docs（文档）、test（测试）、chore（构建/工具）。Scope 为影响的模块或组件名。描述使用英文祈使句（imperative mood），如 "add user login validation" 而非 "added..." 或 "adds..."。每个提交应为原子提交，只做一件事。PR 合并使用 Squash and Merge，保持 main 分支历史整洁。

## 第三章 通用编码规范

### 命名规则

变量和函数使用小驼峰（camelCase，JS/TS）或下划线分隔（snake_case，Python）。类名使用大驼峰（PascalCase）。常量使用全大写下划线分隔（UPPER_SNAKE_CASE）。布尔值变量和函数以 is/has/can/should/will 开头。命名应见名知义，拒绝隐晦缩写（如 ul → userList，gubId → getUserById）。严禁使用拼音命名。中文注释仅限解释特殊业务术语，且须在术语首次出现时标注英文翻译。

### 注释规范

类、接口和公共函数须有 JSDoc/Javadoc/Docstring 风格的文档注释，描述功能职责、参数含义和返回值。复杂业务逻辑须有行内注释解释"为什么这样做"（WHY），而非描述"做了什么"（WHAT，代码本身已表达）。TODO 注释格式：`// TODO(owner): 描述 —— 截止日期`，须包含责任人和计划完成日期。FIXME 注释表示已知缺陷，须关联 Issue 编号。严禁在代码库中保留大段已被注释掉的旧代码——历史代码可从 Git 历史中找回。

### 格式化

前端（JS/TS/TSX）缩进使用 2 个空格；后端（Python）缩进使用 4 个空格。严禁混用空格和 Tab 缩进。每行代码不超过 120 个字符，过长的表达式应在操作符处换行。文件末尾保留一个空行。行尾不得有多余空格（Trailing Whitespace）。所有文件使用 UTF-8 编码。大括号（JS/TS）不另起新行（K&R style）。Python 遵守 PEP 8 的全部格式化要求。

## 第四章 TypeScript/React 规范

React 组件统一使用函数组件 + Hooks，禁止使用 Class 组件。组件文件与组件同名，使用 PascalCase（如 `UserSwitcher.tsx`）。每个组件文件不超过 200 行，超出则拆分子组件或提取自定义 Hook。Props 类型须显式声明为 interface 或 type，禁止使用 `any`。特殊情况须加注释说明原因并获 Reviewer 同意。

状态管理原则：局部状态用 useState，跨组件共享用 Context（轻量）或 Zustand（复杂全局状态）。异步数据请求使用 useEffect + AbortController 或 React Query/SWR 处理竞态条件（Race Condition）和请求取消。严禁在 render 函数中直接修改 state，严禁直接修改 props 对象。

样式使用 Tailwind CSS utility 类，不编写自定义 CSS 文件。仅在 Tailwind 无法满足的复杂动画或第三方库样式覆盖场景下，经团队讨论后可编写少量 CSS Module。响应式设计遵循移动优先原则，使用 sm（640px）/ md（768px）/ lg（1024px）/ xl（1280px）/ 2xl（1536px）断点。

组件测试使用 Vitest + React Testing Library。每个组件至少覆盖正常渲染、边界状态（空数据、loading、error）和关键用户交互。

## 第五章 Python/FastAPI 规范

所有函数参数和返回值须标注类型提示（Type Hints）。Pydantic Model 用于请求和响应序列化，每个字段须有 description 和示例值（example 或 Field examples）。Pydantic v2 优先使用 model_validate 而非 from_orm。

API 端点函数使用 `async def`。IO 密集型操作（数据库查询、HTTP 请求、文件读取）使用异步库（httpx、aiofiles、asyncpg）。CPU 密集型操作（模型推理、嵌入向量计算、大量数据处理）通过 `asyncio.to_thread` 或 `run_in_executor` 放入线程池执行，避免阻塞事件循环。

错误处理：API 返回统一的错误响应格式 `{"detail": "可读的错误描述", "error_code": "ERROR_CODE"}`。业务逻辑层抛出明确的自定义异常（继承自 HTTPException 或自定义 BaseException），由全局异常处理器统一捕获并转换为 HTTP 响应。

## 第六章 代码评审（Code Review）

所有代码合并到 main 前须通过 Code Review。PR 作者须在 PR 描述中填写变更摘要、关联的 Issue/Task 编号和测试说明。至少指定一位 Reviewer，Reviewer 须在 24 工作小时内完成评审。Review Checklist：逻辑正确性、安全漏洞、性能问题、代码风格符合本规范、测试覆盖充分、无遗留调试代码。评审意见须以 Constructive 方式提出，区分 "Must Fix"（必须修改）和 "Nice to Have"（建议优化）。获得所有 Reviewer Approve 且 CI 全绿后方可合并。

## 第七章 自动化检查

CI 流水线在每次 Push 和 PR 时自动执行：Lint（ESLint + Prettier / Ruff + Black）、Type Check（tsc --noEmit / mypy）、Unit Test（Vitest / pytest）、Build（Vite build）。所有检查必须在 10 分钟内完成。检查失败时，CI 通知到 PR 作者的企业微信。紧急修复（hotfix）可经 Tech Lead 审批后跳过部分非关键检查，但 Lint 和 Build 不可跳过。

## 第八章 附则

本规范由研发部技术委员会（Tech Committee）负责修订。规范修订须经技术委员会讨论投票（≥ 2/3 通过），并在全体研发月度会议上宣讲后生效。任何工程师可通过提交 RFC Issue 对规范提出修订建议。
