# RAG 故障排查指南（7 步诊断法 + 本项目工具）

> 来源:Ertas 生产故障指南的诊断检查表 + 本项目的 trace/评测工具结合。
>
> RAG 静默失败——不报错,但返回错的答案。这份指南给你**有序的排查路径**,而不是瞎猜。
>
> **核心原则**:RAG 是"解析→切分→embedding→检索→组装→生成"的链路,**先定位是哪一环坏了,再修**。最常见的错误是"答错了就调 prompt",但根因可能在检索或切分。

---

## 7 步诊断检查表(按顺序执行)

当 RAG 答错/答不全/答非所问时,**按这个顺序排查**,每步有对应的本项目工具:

### 第 1 步:文档在不在存储里?
**问题**:摄取失败比你想的多——文件没被索引,自然检索不到。
**检查**:
```bash
# 看知识库统计(启动 main.py 时会打印)
📊 知识库统计: {'total_documents': X, 'total_chunks': Y, 'departments': {...}}
```
**本项目工具**:`DataPreparationModule.get_statistics()` 看文档数/chunk数/部门分布。如果数字比你预期少,说明有文档没被加载/解析失败。
**常见坑**:PDF 解析失败(_load_pdf 的 except 会跳过)、文件格式不支持。

---

### 第 2 步:块可不可检索?(embedding 是否坏了)
**问题**:用源文档的**精确文本**直接查,如果连这个都召不回,说明 embedding 管道坏了。
**检查**:
```python
# 用文档原文片段直接查向量库
vs.similarity_search_with_score("工作满1年年假5天", k=3)
# 期望:年假制度.md 排第一，相似度高
# 如果失败:embedding 模型没加载/索引和查询用了不同模型
```
**常见坑**:索引和查询用不同 embedding 模型(故障模式8);FAISS 索引文件损坏(删了重建)。

---

### 第 3 步:真实查询能召回吗?(别只测合成查询)
**问题**:你自己编的查询测不出问题——用户实际问法千奇百怪。
**检查**:
```bash
# 跑评测脚本(用真实分布的评测集)
python evaluation/run_eval.py
python evaluation/complex_doc_test.py
```
**本项目工具**:评测集(41条)+ 复杂文档测试。**坑12的教训**:数据集太小会虚高,必须用真实/复杂的查询测。
**常见坑**:用户问"那个咋搞"(模糊) vs 你测"年假申请流程"(标准),召回天差地别。

---

### 第 4 步:块内容完整吗?(切分有没有损坏)⭐ 本项目重点
**问题**:检索到了文档,但 chunk 内容是损坏的(被切断/残缺)。
**检查**:
```python
# 看 chunk 的原始内容
for c in dp.chunks:
    print(c.metadata['source'], c.metadata.get('block_type'), len(c.page_content))
    print(c.page_content[:100])
```
**本项目踩过的真实坑(坑22)**:岗位工资表6行被递归切分散只剩3行,P4/M2 丢失。**用表格感知切分解决**。
**常见坑**:表格被切断、答案跨块、长句被切在逗号。

---

### 第 5 步:上下文到 LLM 了吗?(有没有被截断)
**问题**:检索到了,但拼进 prompt 时被截断或丢失。
**检查**:看 trace 里的"生成"步骤,确认 context 完整。
```
📋 TRACE xxx
  └ 生成  XXXms tok:YYY   ← 看 token 数是否合理
```
**本项目工具**:trace 日志(`data/traces/trace.jsonl`)记录完整输入输出。`_build_context` 有 max_length=3000 截断,要确认没截断关键信息。
**常见坑**:`_build_context` 的 max_length 设太小,把关键 chunk 截掉了。

---

### 第 6 步:LLM 用没用上下文?(检索对但答错=生成问题)
**问题**:上下文完全正确,但 LLM 还是答错——这是**生成问题,不是检索问题**。
**判断**:
- 检索召回对(第3步通过)+ 答错 = **生成环节**坏了 → 调 prompt/换模型
- 检索召回错 = **检索环节**坏了 → 调切分/rerank/阈值
**本项目工具**:看 trace 的"检索"步骤,确认召回的 source 对不对。如果对但答错,查 generation 的 prompt。

---

### 第 7 步:是幻觉还是真不知道?
**问题**:答错了,但分不清是"上下文有但 LLM 没用"(幻觉)还是"上下文真没有"(该拒答)。
**检查**:看召回的 context 里到底有没有答案。
- 有答案 + 答错 = 幻觉 → 加强防幻觉 prompt(故障模式7)
- 没答案 + 硬答 = 阈值没生效 → 检查阈值/rerank(故障模式2)
**本项目对策**:拒答兜底——上下文无答案时强制返回"未找到"。

## 故障 → 排查步骤速查表

| 症状 | 最可能的故障模式 | 先查第几步 |
|---|:--:|:--:|
| "我不知道"(明明有文档) | 1 检索未命中 | 第2/3步 |
| 自信但答错(引错文档) | 2 检索到错的 | 第3步(看召回对不对) |
| 答案是旧的 | 3 过时上下文 | 第1步(看有没有更新) |
| 答案不全(缺一半) | 4 块边界损坏 | 第4步(看chunk完整吗) |
| 笼统不具体 | 5 上下文溢出 | 第5步(看context长度) |
| 泄露了不该看的 | 6 PII/权限 | 查权限过滤 |
| 编造(上下文有却编) | 7 幻觉 | 第6/7步 |
| 召回一直很差 | 8 embedding不匹配 | 第2步 |

## 本项目的调试工具清单

| 工具 | 用途 | 命令/位置 |
|---|---|---|
| **trace 日志** | 看每环节耗时/输入输出/定位瓶颈 | `data/traces/trace.jsonl` + 控制台 |
| **评测脚本** | 量化检索/忠实/相关三维度 | `python evaluation/run_eval.py` |
| **A/B 对比** | 量化不同策略效果 | `python evaluation/ab_compare.py` |
| **复杂文档测试** | 测表格/答案分散场景 | `python evaluation/complex_doc_test.py` |
| **chunk 检查** | 看切分是否损坏 | `dp.chunks` 遍历打印 |
| **向量距离诊断** | 看正负例距离分布定阈值 | `similarity_search_with_score` |

## 💡 面试话术(问"你的 RAG 出问题怎么排查")

> "我不会瞎调 prompt。我有一套 7 步诊断法:先确认文档有没有被索引(摄取失败很常见),再用原文精确查确认 embedding 没坏,然后用真实查询测召回,再看 chunk 内容完不完整(我踩过表格被切断的坑),确认上下文到没到 LLM,最后判断是检索问题还是生成问题。**关键:RAG 是链路,先定位哪一环坏了,再对症下药。最常见的错误是答错就调 prompt,但根因可能在切分或检索。**"

这个回答展示**系统化的 debug 思维**,而不是"遇到问题就重启/换模型"的业余做法。
