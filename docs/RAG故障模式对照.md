> 来源:Ertas Data Suite《RAG 管道故障模式:生产环境调试现场指南》(基于真实生产事件)
>
> **核心观点**:RAG 不崩溃,它"静默失败"——返回答案,但是错的/不全的/污染的。系统看着正常,实际输出垃圾。

> 这份对照表证明:**本项目覆盖了真实生产 RAG 的全部 11 类故障模式**:前 8 类是 RAG 算法层面的检索/生成故障,后 3 类是工程化/可靠性层面的故障(认证缺失、服务单点、质量回归),全部已解。

## 逐个详解(故障 → 本项目怎么应对)

### 故障1:检索未命中(相关文档在但没检索到)

**文章根因**:查询用词和文档不同(问"解雇",文档写"非自愿终止");embedding 模型不懂数据;块太大稀释/太小丢语义。

**本项目对策**:
- 混合检索(向量+BM25+RRF)解决"术语不匹配"——BM25 抓精确词,向量抓语义
- 查询改写(坑5/6)把口语化问题改写成利于检索的术语
- **踩过的真实坑**:中文 BM25 默认空格分词失效(坑5),接 jieba 解决;BM25 在短查询/小语料弱(坑6),靠向量兜底

---

### 故障2:检索到错的(不相关文档得高分)

**文章根因**:向量相似 ≠ 相关性。模板文本(页眉页脚)高分;缺元数据过滤;不同版本混在一起。

**本项目对策**:
- **reranker cross-encoder 精排**(坑10,A/B 实验)——解决"语义像但实际无关"
- 阈值过滤(cosine≥0.3)——丢掉低相关候选
- **A/B 量化证据**:纯向量/混合检索负例拒答率 0%(全错答),加 rerank 后 100%(全拒答),综合准确率 0%→97.3%

---

### 故障3:过时上下文(文档更新但索引没更新)✅ 已用增量索引解决

**文章**:生产中最常见的 RAG 故障,因为"大多数团队建了索引管道,但没建更新管道"。

**本项目对策**:
- **增量索引(content hash 对比,只更新变化的 chunk)**已落地:`build_incremental`(`rag_modules/index_construction.py`)对每个 chunk 计算 content_hash,与已索引版本逐一比对,只重建发生变化的 chunk,未变化的跳过
- 入库阶段(`rag_modules/data_preparation.py`)就把 `content_hash` 写入 chunk 元数据,作为增量比对的依据
- **量化**:文档只改了一段,全量重建需 reindex 全部 N 块;增量索引只重算变化的那几块,I/O 和向量计算量大幅下降
- **面试讲法**:"我识别了这是生产最高频故障,用 content_hash 增量索引解决——只对发生变化的 chunk 重建索引,避免全量重算。"

---

### 故障4:块边界损坏(答案跨块被切断)⭐ 本项目重点解决

**文章**:固定大小分块在任意字符处切断,忽略语义边界。关键段落被一分为二,检索只找到一半。

**本项目对策 + 真实踩坑**:
- **坑22**:测复杂文档发现岗位工资表(6行)被递归切分散,只剩3行,P4/M2 数据丢失
- **解决**:表格感知切分(`_split_text_and_tables`),表格整体不切断
- **量化**:表格完整召回率 50%→100%,P4/M2/工伤/公积金全部恢复可检索

---

### 故障5:上下文溢出(检索太多,LLM 忽略部分)

**文章**:top-k 太大,超出 LLM 有效利用能力,表现"首因/近因偏差"(只引首尾)。

**本项目对策**:
- top_k=3(不是 10-20),rerank 后只传最相关 3 块
- `_build_context` 设 max_length=3000 截断,防上下文过长
- Lost in the Middle:rerank 把最相关的排前面

---

### 故障6:PII/权限泄露(敏感信息泄给无权用户)⭐ 本项目独家护城河

**文章**:含 PII 文档未脱敏就索引,无访问控制层,LLM 忠实包含上下文里的敏感数据。

**本项目对策**:
- **RBAC 权限隔离**(坑7/11):文档入库打 department 元数据,检索前按用户权限过滤候选集
- 用户服务(user_service):user_id → 可见部门映射
- **验证**:HR 员工搜财务文档 → 只返回 HR 文档,财务数据不泄露
- **真实坑**:权限过滤必须"先过滤后检索",不是"先检索后过滤"(坑7)

---

### 故障7:幻觉(上下文对仍编造)

**文章**:LLM 把参数知识和上下文混,参数知识占上风;信任自己胜过上下文。

**本项目对策**:
- **防幻觉 prompt**:强约束"只能依据上下文,无依据就说未找到,严禁编造"
- **拒答兜底**(坑8/9):阈值过滤让负例根本不进生成
- **引用溯源**:答案每个事实标 [文档X-页Y]
- **真实坑**:不加阈值时负例 100% 错答(坑8),加阈值+rerank 后拒答率 100%

---

### 故障8:embedding 不匹配(模型选错)

**文章**:按基准选而非按领域适配;索引和查询用不同模型。

**本项目对策**:
- 索引和查询统一用 bge-small-zh(归一化),绝不混用
- **坑19**:发现 FAISS 默认 L2 不是 cosine,主动优化成 IP/cosine(0~1 直观)
- 认知:"归一化下 L2 和 cosine 单调对应排序等价,但 IP 数值直观、性能略快"

## 工程化故障模式（可靠性维度）

> 上面 8 类是 RAG **算法层面**的检索/生成故障。但在生产里,RAG 系统不只会"算错",还会因为**工程化缺失**而整体不可用或被攻破。下面 3 类是工程化/可靠性层面的故障——同样来自生产教训,本项目均已解决。

### 故障9:身份伪造（认证缺失）✅ 已解

- **症状**:客户端伪造 user_id 越权访问,RBAC 形同虚设
- **文章根因**:API 只做**授权**(user→权限),没做**认证**(验证身份)。知道 user_id 就能冒充任何人,授权建立在被伪造的身份之上。
- **本项目对策**:JWT 认证层(`app/auth.py`)。user_id 从 token 解码,**不是请求体**——客户端无法自行填写。`POST /auth/login` 校验账号密码后签发 JWT token,所有业务接口 `Depends(get_current_user)` 强制校验 token,未通过则 401。
- **对应坑**:坑30
- **状态**:✅已解

---

### 故障10:服务单点（LLM provider 故障）✅ 已解

- **症状**:主 LLM provider 挂了（限流/余额不足/网络中断）,整个 RAG 系统不可用
- **根因**:只有一个 LLM 实例,没有故障切换。`max_retries` 只是重试**同一个** provider,provider 本身挂掉时重试无意义。
- **本项目对策**:`FallbackLLM` 降级链（`rag_modules/llm_fallback.py`）。持有一个 LLM 列表 [primary, fallback1, ...],primary 异常后**自动切换**到备用 provider,熔断后冷却 60s 避免反复打挂掉的节点,请求成功后记忆状态避免抖动。通过 `LLM_FALLBACK_*` 环境变量配置多个备用 provider/key/model/base_url。
- **对应坑**:坑31
- **状态**:✅已解

---

### 故障11:质量回归（无 CI/CD）✅ 已解

- **症状**:改了代码引入回归,但没有自动测试发现,问题流入生产
- **根因**:没有持续集成,测试只能本地手动跑,容易遗漏（"在我机器上能跑"）。
- **本项目对策**:GitHub Actions CI（`.github/workflows/ci.yml`）。push / PR 自动跑 19 项 pytest。测试用 **fake key** 不依赖真实 LLM,既保证可复现又避免 CI 额外消耗 token。
- **对应坑**:坑32
- **状态**:✅已解
