## 一、LLM 接入与配置类

### 坑1:`.env` 里 `LLM_BASE_URL==xxx` 写了两个等号

**现象**:配了 z.ai 地址,但日志显示一直连 `bigmodel.cn`(智谱),报"余额不足 429"。

**排查**:看日志的请求 URL——`POST https://open.bigmodel.cn/...` 暴露了"连的是谁"。再看 `.env`,发现第4行 `LLM_BASE_URL==https://api.z.ai/api/anthropic`(双等号)。

**根因**:`dotenv` 把 `LLM_BASE_URL=`(带等号)当变量名,值是 `https://...`。变量名实际是 `LLM_BASE_URL=` 而非 `LLM_BASE_URL`,所以 `config.py` 读不到 `LLM_BASE_URL`,回退到默认值(智谱)。

**解决**:改成单等号 `LLM_BASE_URL=https://...`。

**💡面试话术**:"调 LLM 报错时,先看日志的请求 URL,它直接暴露连的是哪家。`.env` 语法必须严格——每行 `键=值`,等号两边不加空格,只有一个等号。"

---

### 坑2:GLM 报 429 但实际是"余额不足"(错误码 1113)

**现象**:GLM-4-flash 调用返回 `429 Too Many Requests`,但 message 是"余额不足或无可用资源包"。

**根因**:429 通常意味限流,但智谱的"余额不足"也用 429 状态码 + 错误码 1113 表达。**HTTP 状态码和业务错误码不是一回事**。

**解决**:看 `error.code`(1113)而非 HTTP status(429)。充值或领资源包。

**💡面试话术**:"不能只看 HTTP 状态码判断错误,要看业务 error code。429 在智谱既是限流也是余额不足,我通过 error code 区分。"

---

### 坑3:z.ai 用 Anthropic 协议,不能用 ChatOpenAI

**现象**:z.ai 的端点 `/api/anthropic` 用 OpenAI 协议调,连不上或返回格式不对。

**根因**:z.ai 走 Anthropic 协议(Messages API,`system` 顶层字段、`max_tokens` 必填),跟 OpenAI 协议(`/v1/chat/completions`)字段结构不同。

**解决**:用 `ChatAnthropic` 替代 `ChatOpenAI`,并建 LLM 工厂按 `LLM_PROVIDER` 自动选协议适配器。

**💡面试话术**:"不同供应商协议不同,我建了 LLM 工厂(注册表模式),新增供应商只加一个适配器,不改老代码——供应商解耦。"

---

### 坑4:z.ai base_url 必须 `/` 结尾

**现象**:`https://api.z.ai/api/anthropic`(无尾斜杠)拼路径出错。

**根因**:SDK 拼接 URL 时 `base_url + path`,无尾斜杠会拼成 `.../anthropic/v1/messages` 少了分隔。

**解决**:`https://api.z.ai/api/anthropic/`(带尾斜杠)。

## 二、检索系统类

### 坑5:中文 BM25 默认空格分词,几乎完全失效 ⭐

**现象**:BM25 检索结果不对——查"年假几天"返回了无关文档。

**排查**:直接看 BM25 打分,发现两个文档都得 0 分。原因:`rank_bm25` 默认按空格分词,中文没空格 → 整句被当成一个词 → 跟任何文档都匹配不上。

**根因**:`BM25Retriever.from_documents()` 默认 `preprocess_func` 是英文分词(空格切),中文场景完全失效。

**解决**:传 `preprocess_func=chinese_tokenizer`(jieba 分词)。

```python
self.bm25_retriever = BM25Retriever.from_documents(
    self.chunks, k=5,
    preprocess_func=chinese_tokenizer,  # 中文必须分词
)
```

**💡面试话术(高频题)**:"BM25 在中文上默认按空格分词,整句变一个词,完全失效。我接了 jieba 中文分词。**BM25 是算法不是工具**——不一定用 ES,rank_bm25 纯 Python 够用,百万级文档才上 ES。"

---

### 坑6:BM25 在小语料 + 短查询上效果差 ⭐

**现象**:即便加了 jieba 分词,查"年假几天"BM25 还是得 0 分,返回错文档。

**排查**:看分词结果——`"年假几天"→['年','假','几天']`,文档里是"1年""5天",**字面不完全一致**,BM25 关键词匹配匹配不到。

**根因**:BM25 是字面关键词匹配,不懂变形/语义。短查询词少,IDF 区分度弱,小语料更严重。

**认知升华**:**这恰恰证明了混合检索的必要性**——BM25 搞不定的语义匹配,向量检索兜底。两者 RRF 融合,互补而非替代。

**💡面试话术**:"BM25 在短查询/小语料上弱,不懂语义变形。这正是为什么要做混合检索——向量负责语义,BM25 负责精确词,RRF 融合。单独任何一个都不够。"

---

### 坑7:权限过滤"先检索后过滤"会召回不足 ⭐⭐

**现象**:有权限的用户搜自己部门的文档,有时搜不到。

**根因**:C8 原实现是"先 hybrid_search 召回 top5,再按 metadata 过滤"。如果这 5 个都不在权限内,过滤后变空——**有权限的用户反而搜不到自己的文档**。

**解决**:改成"先过滤 chunks 子集(权限边界),再在子集上检索"。

```python
# 错误:先检索后过滤(topK 召回后过滤可能为空)
docs = self.hybrid_search(query, top_k)  # 召回5个
docs = [d for d in docs if filter(d)]     # 过滤后可能空

# 正确:先过滤后检索(在权限子集上检索)
allowed = [d for d in chunks if filter(d)]  # 先限定权限集
docs = retrieve(query, allowed)             # 再检索
```

**💡面试话术**:"权限过滤必须在检索前做,不是检索后做。否则 topK 召回不足时,有权限的用户反而搜不到自己的文档。我在文档入库时打 department 元数据,检索前先按 RBAC 过滤候选集。"

---

### 坑8:不加阈值,topK 强制返回导致负例无法拒答 ⭐⭐

**现象**:评测发现负例(知识库无答案的问题)无法拒答,系统硬返回了无关文档。context_recall 虚高到 1.0。

**根因**:`hybrid_search` 的 topK=3 意味着"无论如何返回 3 个",即使相关性极低也返回 → 负例无法拒答 → 评测虚高 + 幻觉。

**解决**:加向量距离阈值过滤(距离 > 阈值则丢弃)。实测:正例相关文档距离 ~0.7,负例无关文档距离 ~1.18,阈值设 1.0。

---

### 坑9:只过滤向量路不够,BM25 又把负例捞回来 ⭐

**现象**:加了向量阈值过滤,但负例还是返回了文档。

**排查**:负例在向量路被过滤了,但 BM25(关键词匹配)依然返回了文档(它没阈值概念),RRF 融合后又被捞回。

**根因**:只过滤了一路,另一路有漏洞。

**解决**:**向量路是语义相关性的主要判据,向量路过滤后为空,即使 BM25 有命中也直接拒答**。BM25 只是"向量已召回基础上的补充",不能单独决定返回。

**💡面试话术**:"检索的每一路都要受控。我以向量路为主判据——向量路无相关结果就拒答,即使 BM25 有关键词命中。否则负例会从 BM25 漏回来。"

---

### 坑10:固定阈值无法完美区分正负例(距离重叠)→ 必须 rerank ⭐⭐⭐

**现象**:阈值设 1.0,负例拒答了,但 2 个正例(借用逾期/高值物品)被误杀。context_recall 从 1.0 掉到 0.95。

**排查**:看距离分布——
```
正例距离: 0.70 ~ 1.04  (借用逾期 1.04 最高)
负例距离: 0.89 ~ 1.18  (公司年会 0.89 最低)
```
**两者重叠!"公司年会"(负例,0.89)比"借用逾期"(正例,1.04)距离更近。** 无论阈值设多少,要么漏正例要么收负例。

**根因**:**向量检索(bi-encoder)的本质局限**——它分不清"语义相近但实际无关"。"公司年会"和"年假"在向量空间近(都是HR相关词),但实际无关。

**解决**:加 reranker(cross-encoder)精排。标准两阶段:
```
向量宽松召回(阈值1.3,不漏正例) + BM25 → RRF融合 → rerank精排(去噪+排序) → 按rerank分数过滤+截断topK
```
rerank 实测分数:相关文档 0.97,无关文档 0.0001——完美区分。

**最终结果**:context_recall 从 0.95(误杀正例)→ 1.0(全召回),负例拒答准确率 100%。

**💡面试话术(最硬核的压轴题)**:"纯向量检索有距离重叠问题——正负例的距离区间交叉,固定阈值无法完美区分。这是 bi-encoder 的本质局限。我加了 bge-reranker cross-encoder 精排:向量+BM25 宽松召回保证不漏,rerank 精排保证去噪。**context_recall 从纯阈值 0.95 提到 1.0,负例拒答 100%**。这就是为什么企业 RAG 必须有 rerank。"

### 坑19:FAISS 默认返回 L2 距离不是 cosine,且能 >1、方向相反 ⭐⭐

**现象**:调试阈值时发现向量"距离"能超过 1(正例 ~0.7,负例 ~1.18),直觉上"相似度"应该 0~1,困惑"距离>1 是怎么回事"。

**排查**:看 FAISS 文档——`similarity_search_with_score` 默认返回的是 **L2 欧氏距离**(两个向量在空间的直线距离),不是 cosine 余弦相似度。bge 向量归一化后(长度=1),L2 距离范围是 **0~2**(不是 0~1)。

**根因**:很多人(含教程)把"向量相似度"默认当 cosine(0~1,越大越像),但 FAISS 默认是 L2。两者完全不同:

| | L2 欧氏距离 | cosine 余弦相似度 |
|---|---|---|
| 含义 | 空间直线距离 | 方向夹角余弦 |
| 范围 | **0~2**(归一化后) | -1~1(通常 0~1) |
| 越相似 | **越小**(0=相同) | **越大**(1=相同) |
| 能否 >1 | **能!** | 不能 |

数学关系(归一化向量下):**L2 = √(2 - 2×cosine)**。换算:
- cosine=1.0 → L2=0
- cosine=0.5 → L2=1.0(你的阈值点)
- cosine=0.0 → L2=1.414
- 你的正例 L2≈0.7 → cosine≈0.755
- 你的负例 L2≈1.18 → cosine≈0.304

**关键坑**:L2 和 cosine **方向相反**(L2 越小越好,cosine 越大越好),过滤/排序时搞反就全错。

**解决(主动优化,非被动修 bug)**:
经分析,语义检索场景应该用 cosine——它只比方向(语义),不被向量长度噪声干扰,且数值直观(0~1)。归一化下 L2 与 cosine 单调对应、排序等价,但 cosine/IP 更优。于是主动把索引从 L2(`IndexFlatL2`)改成内积(`IndexFlatIP`,归一化下=cosine):

```python
# 优化前:FAISS 默认 L2(距离 0~2,越小越好,阈值要换算)
store = FAISS.from_documents(chunks, embeddings)

# 优化后:手动用 IndexFlatIP(cosine 0~1,越大越好,阈值直观)
import faiss
store = FAISS.from_documents(chunks, embeddings)  # 先建
vectors = store.index.reconstruct_n(0, store.index.ntotal)
store.index = faiss.IndexFlatIP(store.index.d)   # 换成内积
store.index.add(vectors)                          # 重新加归一化向量
```

改完后阈值从"距离≤1.3"(要换算)→"相似度≥0.3"(直观),正例 cosine≈0.65、负例 cosine≈0.41,一眼能看懂。评测验证 context_recall 仍 1.0,功能正常。

**💡面试话术(高频题)**:"语义检索应该用 cosine——它只比方向不受长度噪声干扰。但 FAISS 原生不支持 cosine,业界做法是**向量先归一化再用内积 IP**,数学上等价。我一开始用 FAISS 默认的 L2(归一化下和 cosine 单调对应,排序等价),后来主动优化成 IP——数值直观(0~1 相似度)、阈值好定、性能略快。**注意 L2 越小越好、cosine 越大越好,方向相反,过滤排序不能搞反。**"

**这个坑的双重价值**:① 一开始被 L2>1 困惑,查文档搞懂了原理(真实学习过程)② 搞懂后主动从 L2 优化成 cosine(真实工程改进)。能讲出"发现默认配置不最优→理解原理→主动优化"的完整闭环,证明是主动思考的工程师,不是被动调包。

### 坑22:复杂文档的表格被递归切分散,6行表只装下3行 ⭐⭐⭐

**现象**:用复杂文档(薪酬办法,含2个嵌套表格)测召回率,检索层面 100% 召回——看着完美。但深查发现:**表格被切分边界切断了**。

**排查**:看薪酬文档的 chunk 分布:
```
薪酬文档切成 4 个 chunk:
  chunk1: 岗位工资表 —— 只有 P1/P2/P3 三行！P4/M1/M2 去哪了？
  chunk3: 社保公积金表 —— 只到失业保险！工伤保险/公积金去哪了？
```
原本 6 行的岗位工资表,chunk1(500字限制)只装得下表头+3行,**P4/M1/M2 这3行被切到了下一个 chunk,但下一个 chunk 是绩效奖金文本,表格彻底断了**。

**根因**:`RecursiveCharacterTextSplitter` 按"段落→句子→字符"递归切分,**它不认识"表格"是一个整体**——表格被当成普通文本按字数切断。这导致:
- 问"P4 基本工资"→ 检索到的是残缺表格(只有P1-P3)→ 模型答不出或答错
- 表格的关联性(同一张表的行)被破坏

**为什么之前的"100%召回率"是假象**:
我之前的测试只验证了"**有没有召回薪酬文档**",没验证"**召回的 chunk 里表格完不完整**"。检索召回了正确的文档,但那个 chunk 里的表格是残缺的——检索层 100%,但生成层拿不到完整表格,会答错或答不全。

**真实复杂文档的核心难点(比 demo 难得多)**:
| 问题 | demo 文档 | 真实文档 |
|---|---|---|
| 表格 | 无 / 小 | 多个、嵌套、跨页 |
| 切分 | 文本均匀 | 表格/图片打断文本流 |
| 答案位置 | 单点 | 散落在表格多行 |

**解决方向(已识别,待实施)**:
1. **表格感知切分**:识别 markdown 表格边界(`|...|`),表格作为一个整体不切断
2. **语义切分**:用 embedding 判断语义边界,而非纯字数
3. **父子检索**:小块检索命中 → 回溯包含完整表格的父块(已实现父子机制,但要保证父块含完整表格)
4. **表格结构化解析**:PDF 表格用 Camelot/Unstructured 转成结构化数据,而非当文本切

**实际解决(已实施):表格感知切分**

实现了 `_split_text_and_tables`——先把文档按表格边界拆成「文本块 + 表格块」序列,表格块整体不切,文本块照常递归切分:
```python
def _split_text_and_tables(self, content):
    # 识别连续的 |...| 行作为表格整体，其余为文本
    # 表格 → 整体一个 chunk（不切断）
    # 文本 → 递归切分
```

**优化前后对比(真实量化)**:
| 指标 | 优化前(递归切分) | 优化后(表格感知) |
|---|---|---|
| 岗位工资表 | 只剩3行(P1/P2/P3) | 完整7行(P1~M2) |
| P4 专家数据 | ❌ 丢失,查不到 | ✅ 完整可检索 |
| M2 经理数据 | ❌ 丢失,查不到 | ✅ 完整可检索 |
| 社保公积金表 | 只剩3行(养老/医疗/失业) | 完整6行 |
| 工伤/公积金 | ❌ 丢失 | ✅ 完整可检索 |
| 表格完整召回率 | ~50%(3/6行) | **100%(6/6行)** |

验证:问"P4专家基本工资"→ 召回完整7行表格,准确找到 P4=25000;问"M2经理岗位津贴"→ 找到 M2=8000。**之前丢失的数据全部恢复。**

**💡面试话术(真实复杂文档的硬核认知)**:
"我一开始在简单文档上测召回率 100%,但换了含表格的复杂文档,深查发现**表格被递归切分散了**——6行岗位工资表只装下3行,P4/M2 数据丢了。这说明'召回率100%'是假象,它只验证了检索到文档,没验证召回的 chunk 表格完整。**真实复杂 RAG 的核心难点不是检索,是切分**。我实现了表格感知切分——识别 `|` 表格边界,表格整体不切断,文本块照常递归。优化后表格完整召回率从 ~50%(3/6行)提升到 100%,之前丢失的 P4/M2/工伤/公积金数据全部恢复可检索。"

**这个坑的元价值**:它揭示了"评测指标本身可能误导"。召回率 100% 不等于系统好——还要看召回的内容是否完整、可用。**这正是为什么评估要三维度(检索+忠实+相关),且要人工抽检 bad case,不能只看数字。**

## 四、评测类

### 坑12:评测数据集太小导致指标虚高 ⭐

**现象**:2 个文档时,context_recall/faithfulness/answer_relevancy 全是 1.0 满分——不真实。

**根因**:全库只有 2 个文档,任何查询都能把这俩召回,召回必然 1.0。**就像考试只考你会的 2 道题**。

**认知**:**评测集太小,指标必然失真**。必须扩大到几十篇文档 + 上百条评测集,指标才有区分度。

**💡面试话术**:"我一开始评测集太小(2文档),指标虚高到 1.0。扩到 5 文档 41 评测集后,立刻暴露了真实问题(负例无法拒答)。评测集规模直接决定指标可信度。"

### 坑13:LLM 裁判太宽松,负例打满分 ⭐

**现象**:负例(股票期权)该拒答,但 LLM 裁判给了 faithfulness=1。

### 坑20:trace 发现查询改写占 80% 延迟,简单问题不该走 LLM ⭐⭐⭐

**现象**:trace 显示一个简单事实查询"密码多少天更换一次"总耗时 10.7 秒,其中:
```
意图路由  2903.9ms   ← 花了 3 秒只为判断"这是检索类"
查询改写  6355.1ms   ← 花了 6 秒！
检索       17.5ms    ← 实际检索只要 0.017 秒
生成     1457.9ms
```
**意图路由 + 查询改写占了 8.6 秒(80%),而真正干活的检索只要 0.017 秒**。

**根因**:所有问题无差别走 LLM 路由 + 改写。但像"密码多少天更换一次"这种**明确问题,根本不需要改写**——直接拿原 query 检索就行。改写是为"模糊/口语化"问题准备的(如"那个怎么搞")。

**解决(短路路由)**:判断问题是否明确,明确的直接跳过 LLM 改写:
```python
def is_simple_query(query):
    # 过短/纯口语 → 需改写
    if len(query) < 4: return False
    if any(v in query for v in ["那个","怎么搞","帮我看看"]): return False
    return True  # 含具体名词 → 明确，跳过改写

if is_simple_query(question):
    rewritten = question          # 短路：直接用原 query
else:
    rewritten = llm_query_rewrite(question)  # 模糊才走 LLM
```

**效果**:查询改写从 6355ms → **0ms**(明确问题),延迟从 10.7s → 6.3s(-41%)。

**💡面试话术**:"我通过 trace 发现,80% 延迟花在意图路由和查询改写上,而实际检索只要 0.017 秒。但明确问题根本不需要改写。我加了短路路由——明确问题跳过 LLM 改写,只有模糊/口语化问题才走改写。延迟降了 41%。**可观测不是为了好看,是为了定位瓶颈、数据驱动优化。**"

### 坑21:语义缓存阈值难调(0.92 太严,合理改写命中不了)⭐⭐

**现象**:加了语义缓存(相似问题秒回),但"密码多少天更换一次"存了缓存后,问"密码更换周期"没命中——还是走完整链路。

**排查**:实测这两个 query 的 cosine 相似度:
```
"密码多少天更换一次" vs "密码更换周期"   = 0.807   ← 阈值 0.92，没命中！
"密码多少天更换一次" vs "密码多久换一次"  = 0.942   ← 这个能命中
"密码多少天更换一次" vs "年假几天"       = 0.519   ← 不同主题
"密码多少天更换一次" vs "住宿费报销"     = 0.371   ← 不同主题
```

**根因**:阈值设 0.92 太严——"密码更换周期"这种合理改写(0.807)都命中不了。但同主题问题(0.8~0.94)和不同主题(0.37~0.52)有明显分界,阈值可以放到 0.78。

**解决**:阈值 0.92 → 0.78。实测后:
- 精确缓存(完全相同问题):**0.1ms** 命中(从 10000ms)
- 语义缓存(同义改写):**28ms** 命中(从 10000ms)

**关键认知**:语义缓存阈值是**精度/误命中权衡**——
- 太严(0.92)= 等于没有缓存(合理改写命中不了)
- 太松(0.5)= 不同问题返回相同答案(答非所问)
- 必须实测相似度分布,找到合理分界点(0.78)

**💡面试话术**:"语义缓存不是加就行,阈值是关键。我实测了同义改写(0.80~0.94)和不同主题(0.37~0.52)的相似度分布,把阈值从默认 0.92 调到 0.78——既能让合理改写命中,又不会误命中不同主题。**缓存让高频问题从 10 秒降到 0.1ms(10万倍),但前提是阈值调对。**"

**性能提升总结(坑20+21)**:
```

高频问题(缓存): 10000ms → 0.1ms(精确) / 28ms(语义)
```

### 坑23:语义缓存误命中——"p2工资"≠"p1和p2相差多少" ⭐⭐⭐

**现象**:用户问"p1 和 P2 工资相差多少"(需比较计算),系统却返回了"p2 基本工资为12000"(单值事实)——**答非所问**。

**排查**:看 trace:
```
❓ p1 和 P2 工资相差多少
⚡ 缓存命中(semantic)，sim=0.854，直接返回
✍️ 回答: P2（中级）的基本工资为12000元/月   ← 返回的是"p2工资多少"的缓存答案！
```
"p1 和 P2 工资相差多少" 和 "p2 基本工资多少" 的语义相似度 0.854,命中了缓存(阈值 0.78)。但这是**两个完全不同的问题**——一个要"差值"(需计算),一个要"单值"(直接事实)。

**根因**:坑21 调的语义阈值 0.78 **太松**了——它让"相似主题但不同问题"也匹配上。语义相似(都是"工资P2")≠ 问题是同一个(差值 vs 单值)。**语义缓存的最大风险就是误命中——返回的不是用户问的。**

1. **长度差异**:两 query 长度差 >40% → 可能不同问题
2. **实体匹配**:新 query 独有的实体(P1)若不在 cached_q 里 → 问的不是同一个对象
3. **比较意图**:含"相差/比较/差多少"的比较类问题,不能命中单值类的缓存

```python
# 修复后："p1和P2相差多少" 语义相似度 0.854，但被校验否决
[缓存-语义命中被否决] sim=0.854 但实体/长度校验未过: 
  p1 和 P2 工资相差多少 vs p2 基本工资多少
```

**💡面试话术(缓存设计的真实难点)**:
"语义缓存不是加就行,误命中比没缓存更糟——会答非所问。我一开始阈值设 0.78,结果'p1和p2相差多少'命中了'p2工资'的缓存,返回了错误的单值。这暴露了'语义相似≠问题相同'。我加了三重校验——长度差异、实体匹配、比较意图——任一不过就否决命中。**缓存的核心矛盾:命中要准(不误命中),又要松(能命中同义改写),这两者天然冲突,必须多重校验来平衡。**"

### 坑25:缓存数字误命中——"满1年"返回了"满3年"的答案 ⭐⭐⭐

**现象**:用户问"工作满1年年假几天",系统返回"工作满3年不满5年的年假为10天"——**事实性错误**!

```
❓ 工作满3年年假几天  → 回答"满3年10天" ✅
❓ 工作满1年年假几天  ← 用户问满1年
```

### 坑26:全局缓存跨权限泄露——缓存层绕过了 RBAC ⚠️⚠️

**现象**:张三(HR)问"年假几天"存了缓存,李四(财务)问同样问题**直接从缓存返回了 HR 的答案**——数据跨权限泄露!虽然检索层做了 RBAC 隔离(财务看不到 HR 文档),但缓存层是全局的,绕过了权限控制。

**根因**:缓存只按"问题"存,不区分用户权限。`_hash(query)` 只用 query 生成 key,张三和李四问同一问题时 key 相同 → 命中同一条缓存 → 权限被绕过。**缓存层和检索层的权限是两套,缓存没有继承检索的权限控制。**

**解决:缓存带权限签名**,`_hash` 改为 `_hash(query, perm_sig)`:
```python
# 修复前(全局,不区分用户权限):
def _hash(self, query):
    return md5(query)  # 张三和李四的 hash 相同 → 缓存串

# 修复后(权限隔离):
def _hash(self, query, perm_sig):
    return md5(f"{query}|{perm_sig}")  # key 含权限签名 → HR/财务各自缓存
```
语义匹配也只在同权限签名的缓存条目里匹配,不同权限不串。

**验证**:张三(HR)存"年假几天"答案,李四(财务)问同样问题 → ❌ 不命中(权限签名不同),各自走完整链路。

**💡面试话术(缓存层的安全设计)**:
"缓存和权限必须一体化设计,否则缓存会绕过权限。我做了 RBAC 检索隔离,但一开始缓存是全局的——张三(HR)问了答案,李四(财务)直接从缓存拿到,权限白做了。我把权限签名(用户可见部门集合)嵌入缓存的 key——不同权限问同一问题各自缓存、各自命中,语义匹配也只搜同权限条目。**任何带权限的系统,缓存层必须和权限层同构,否则缓存就是后门。**"

### 坑27:增量索引首次实现——存了 hash 但未保存(early return bug) ⭐

**现象**:实现了增量索引(content_hash 对比),但第二次运行时仍"0 块不变,10 块需重新 embedding",增量完全失效。

**排查**:看 `build_incremental` 代码——第4步当 `existing` 为 None(首次)时,调 `build_vector_index` 后 `return` 了,跳过了第5步的 `hash_file.write_text()`。**首次运行永远不存 hash,第二次没得对比,所以增量永远不生效。**

**根因**:写代码时没注意控制流——`return` 把后面的保存步骤跳过了。

**解决**:去掉 `return`,让首次构建也走完第5步保存 hash:
```python
# 修复前
if not existing:
    self.build_vector_index(chunks)
    return self.vectorstore  # ← 跳过保存 hash!

# 修复后
if not existing:
    self.build_vector_index(chunks)
    # 不 return,继续往下保存 hash
```
修复后:第2次运行显示"增量更新: 全部 10 块未变化,跳过重建" ✅

### 坑29:增量索引更新后缓存未同步——返回旧答案 ⭐⭐⭐

**现象**:文档更新后,增量索引正确重建了 embedding,但用户问同样的问题仍返回**修改前的答案**——索引新了,缓存旧了。

**场景模拟**:年假从"5天"改成"7天"→增量索引检测到变化重新 embedding → 用户问"年假几天"→ 缓存命中 → 返回"5天"(旧答案)。

**根因**:缓存和索引是**两套独立系统**,索引的增量更新不会自动通知缓存失效。这是经典的**缓存一致性问题**——两个数据层之间有状态依赖,但没有同步机制。

**解决**:`build_incremental` 返回 `(vectorstore, has_changes)` 元组。`has_changes=True` 时,`main.py` 自动调用 `cache.clear()` 清空所有缓存。下次查询全部走完整链路,生成新答案并存缓存。

```python
# main.py
vectorstore, has_changes = self.index_module.build_incremental(chunks)
if has_changes and self.cache:
    self.cache.clear()  # 文档变了 → 缓存全部失效
```

**设计权衡(面试可讲)**:这里选择"**全量清缓存**"而非"精准失效只清受影响的条目",原因:
- 缓存条数少(内存级),全清成本极低
- 精准失效需要追踪"每条缓存依赖哪些文档",实现复杂,维护成本高
- 文档更新不频繁,全清不影响命中率
- **简单正确的方案优于复杂但不稳的方案**

**💡面试话术**:
"增量索引解决索引更新的问题,但引入了新的坑——缓存没同步。文档变了,索引更新了,但缓存命中返回的仍是旧答案。这是经典的缓存一致性。我的方案是增量索引返回变化标记,有变化时自动全量清缓存——简单、正确、零维护。面试官可能会追问'为什么不精准失效只清受影响的',我的回答是缓存条数少且文档更新不频繁,全清的代价可以忽略,但精准失效的实现复杂度和出错概率高得多。**工程取舍的核心:用最简单的方案解决正确性问题。**"

```
高频问题(缓存): 10000ms → 0.1ms(精确/语义，带四重误命中+权限隔离+索引同步校验)
```

### 坑33：增量索引 add_documents 导致僵尸向量——检索召回重复内容 🔴 P0

**现象**：文档编辑后，检索返回同一 chunk 的多个版本（新旧内容同时出现），用户看到几乎一样的引用来源。

**排查**：看 `build_incremental` 的修复路径——当 `existing` 不为 None 时，调 `existing.add_documents(new_chunks)` 追加新 chunk。但 FAISS `IndexFlatIP` **不支持删除向量**，编辑文件后旧版本的 chunk 永远留在索引里。
```
文件A 原来 5 个 chunk (index 0-4) → FAISS 有 5 个向量
文件A 编辑后变 7 个 chunk (index 0-6) → 7 个全标记"new"
→ existing.add_documents(new_chunks) → 只追加，不删旧的 5 个
→ FAISS 现在 12 个向量，其中 5 个是僵尸数据
```

**根因**：`IndexFlatIP` 不支持 `remove_ids`，增量追加在新旧版本文件之间会产生僵尸向量。`content_hash` 去重只在 chunk 级别生效，但新旧 chunk 的 `chunk_id` 不同（每次 `chunk_documents()` 生成新 UUID），导致同一语义内容以多个向量存在。

**解决**：检测到变化时全量重建索引（`self.build_vector_index(chunks)`），而非增量追加。
```python
# 修复前
if existing:
    existing.add_documents(new_chunks)  # 僵尸向量！

# 修复后
if new_chunks:
    self.build_vector_index(chunks)  # 全量重建，干净无残留
```

**💡面试话术**："增量索引有个隐藏陷阱——FAISS IndexFlatIP 不支持删除向量。编辑文件后旧版本 chunk 永远留在索引里，导致检索召回重复内容。我改为检测到变化时全量重建索引——牺牲一点 embedding 时间换正确性。"

### 坑35：Reranker 前未去重——相似块浪费算力且返回重复 🟡 P1

**现象**：PDF 页眉/页脚在每页出现，切分后产生多个几乎相同的 chunk。这些重复块都进入 Reranker（cross-encoder 推理慢），且最终结果看起来全是同一句话的不同页码。

**排查**：去重只在 `hybrid_search()` 做了——`_rbac_subindex_search` 和 `metadata_filtered_search` 完全没有去重。且精确 `content_hash` 去重只能去完全相同的内容，对 99% 相似的页眉无效。

**解决**：用 MMR（Maximal Marginal Relevance）替换简单去重，覆盖全部 3 个检索路径：
```python
def _mmr_dedup(docs, lam=0.7, sim_threshold=0.85):
    # 第一步：content_hash 精确去重（完全相同的只保留最高分）
    # 第二步：MMR 多样性去重
    #   score = λ × relevance - (1-λ) × max_jaccard(候选, 已选中)
    #   Jaccard 用字符 3-gram，无需分词，中文友好
```

**效果**：PDF 页眉重复块消失，Reranker 输入候选减少 ~30%。

**💡面试话术**："去重不是简单按 chunk_id 排重——PDF 每页页眉是不同的 chunk_id 但内容 99% 相似。我用 MMR 算法做多样性去重，关键是覆盖所有检索路径——我一开始只在 hybrid_search 做了，review 被抓出 RBAC 路径没做。"

### 坑36：PDF 扫描件无法提取文字——PyMuPDF 对图片型 PDF 返回空 🟢 P2

**现象**：扫描版合同 PDF（图片型，无文字层）→ `page.get_text("text")` 返回空 → 整页被跳过 → 知识库里没有这份文档。

**根因**：PyMuPDF 只能提取嵌入式文字层，扫描件没有文字层。

**解决**：`_load_pdf()` 加 OCR 兜底：
```python
def _ocr_page(page, page_num, file_path, lang="chi_sim+eng"):
    try:
        import pytesseract; from PIL import Image
    except ImportError:
        return ""  # 优雅降级
    pix = page.get_pixmap(dpi=300)
    img = Image.frombytes("RGB", (pix.width, pix.height), pix.samples)
    return pytesseract.image_to_string(img, lang=lang)
```
依赖 pytesseract + Pillow（pip）+ 系统安装 Tesseract OCR。未安装时优雅降级。

**💡面试话术**："企业文档不只有 Office 导出的 PDF，还有很多扫描件。OCR 兜底是刚需——我用 Tesseract + 300dpi 中英文混合识别，关键是优雅降级设计，OCR 没装也不影响正常 PDF 解析。"

### 坑37：Trace token 全为零 + 成本算错价格 ⭐

**现象**：trace 输出 `tokens:输入0 输出0 共0 | ≈$0.0`——完全不知道每次问答消耗了多少 token。即使有 token 数，价格也是错的（写死 GLM-4.6 定价）。

**排查**：三个独立问题交织：
1. 流式路径硬编码 `{"prompt": 0, "completion": 0, "total": 0}`
2. `_extract_tokens()` 只认一种 `usage_metadata` 格式，不同 LangChain 版本可能返回 pydantic model
3. 成本计算写死 GLM-4.6 价格 `$0.07/$0.29 per M token`

**解决**：
```python
# ① 流式：字符估算（中英文 ~2 char/tok）
est_tokens = max(1, len(answer) // 2)

# ② _extract_tokens：5 种格式 fallback
#    usage_metadata(dict) → token_usage → usage(pydantic) → 字符估算

# ③ 按 provider 查表动态定价
_PRICE_PER_TOKEN = {"deepseek": (0.00000014, 0.00000028), ...}

# ④ DeepSeek 流式真实 usage
model_kwargs={"stream_options": {"include_usage": True}}
```

**效果**：`≈$5.4e-05` → `≈$0.000054`（DeepSeek 比 GLM-4.6 便宜 4 倍）。

**💡面试话术**："可观测不只是能看 trace，更要能量化成本。我做了三层修复：流式字符估算、多格式 token 提取、按 provider 动态定价。关键是 DeepSeek 的 `stream_options=include_usage` 能在流式最后一帧返回真实 usage。"

### 坑30：客户端传 user_id 就能越权（认证缺失）⚠️⚠️

**现象**：评审模拟攻击——`POST /api/v1/chat` 传 `{"user_id":"admin"}` 就能以管理员身份查询，获取全库文档。RBAC 形同虚设。

**排查**：看 `api.py` 的 `ChatRequest` 模型，`user_id` 是请求体字段，客户端传什么就是什么。没有任何 token/JWT 校验。

**根因**：权限系统只有"授权"层（user_id→departments），没有"认证"层（验证 user_id 是真的）。等于"知道门牌号就能进"。

**解决**：新增 `app/auth.py`（JWT 签发/校验），新增 `POST /auth/login` 端点。所有业务接口加 `Depends(get_current_user)`。请求体里的 `user_id` 字段全部删除——user_id 只从 JWT 解码。

**量化**：攻击面从"传一个字段越权"→"必须持有有效 JWT 签名"，伪造成本从 0 提升到需要破解 HS256 密钥。

**💡面试话术**："评审指出我的 RBAC 缺认证层——客户端传 user_id=admin 就能越权。这是典型的'授权做了认证没做'。我用 JWT 补上了认证层，user_id 只从 token 解码，请求体不再有 user_id 字段。三层安全：认证(JWT)→授权(user_service)→过滤(retrieval)。"

### 坑31：LLM 单点故障（无降级链）⭐⭐

**现象**：评审追问——如果 z.ai/DeepSeek 挂了 30 分钟，用户就干等着？`max_retries=3` 只是重试同一个 provider，不是切到备用 provider。

**排查**：看 `llm_factory.py` 的 `create_llm`，每次只创建一个 LLM 实例。`generation_integration.py` 的 `setup_llm` 也只持有一个 `self.llm`。没有 fallback chain。

**根因**：工厂模式解决了"多 provider 接入"，但没解决"provider 故障切换"。这是两个不同的问题。

**解决**：新增 `llm_fallback.py` 的 `FallbackLLM` 类。持有 LLM 列表，invoke 时依次尝试。带熔断冷却（失败 provider 60s 内不重试）和成功记忆（上次成功的优先用）。无降级配置时退回单 LLM，零额外开销。

**量化**：可用性从"单 provider 挂=全站不可用"→"主备都挂才不可用"。熔断冷却避免"每请求都白试 primary 再切"的延迟浪费。

**💡面试话术**："评审问 LLM 挂了怎么办，我原来只有 max_retries（重试同一个 provider）。我加了 FallbackLLM 降级链——primary 挂了自动切备用，带熔断冷却避免抖动。关键是组合模式不是继承，duck-type 兼容 LangChain Runnable，拿得到 usage_metadata。"

## 附:A/B 对比实验(rerank 价值的最硬证据)

> 对应坑8/9/10。用同一评测集(41条:37正例+4负例)对比三种检索策略,用真实数据证明 rerank 的必要性。
> 实验脚本:`evaluation/ab_compare.py`。

### 对比表

| 策略 | 正例召回率 | 负例拒答率 | 综合分(召回×拒答) |
|---|---:|---:|---:|
| A 纯向量(cosine) | 100% | **0%** | 0% |
| B 混合检索(向量+BM25+RRF) | 100% | **0%** | 0% |
| **C 混合+rerank** | 97.3% | **100%** | **97.3%** |

### 三个关键洞察(面试讲点)

**洞察1:没 rerank,负例 100% 答错(A/B 负例拒答都是 0%)**
4 个负例(股票/公积金/年会/团建)被全部错误返回文档。**没有精排,系统无法识别"语义相近但实际无关"的文档,会硬答导致幻觉。** 混合检索(B)也救不了——它只提升召回,不提升精度。

**洞察2:rerank 让负例拒答率从 0% → 100%(核心价值)**
加 rerank 后,4 个负例全部正确拒答。这是 rerank 最硬的价值——**消除幻觉**。

**洞察3:rerank 牺牲 2.7% 正例召回(真实精度/召回权衡)** ⭐
正例召回从 100% → 97.3%(误杀 1 个边界正例)。这是真实的工程权衡——**用 2.7% 召回换 100% 负例拒答,值得,因为幻觉比漏答危险得多**(企业场景里,答错比答不出严重)。

### 简历可直接写

> "通过 A/B 对比实验量化 rerank 价值:纯向量和混合检索的负例拒答率均为 0%(全部错误作答导致幻觉),加入 bge-reranker 精排后,负例拒答率提升至 100%,综合准确率(召回×拒答)从 0% 提升至 97.3%。代价是正例召回率下降 2.7%(精度/召回权衡),但企业场景下消除幻觉的价值远高于这点召回损失。"

> "我做了 A/B 对比。纯向量和混合检索的正例召回都是 100%,但**负例拒答率都是 0%**——无关问题全被硬答了。加 bge-reranker 后,负例拒答率到 100%,综合准确率 97.3%。代价是正例召回掉 2.7%,但这是值得的权衡——企业场景答错比漏答危险。**rerank 解决的不是'召回更多',而是'判断真假相关,消除幻觉'**。"
