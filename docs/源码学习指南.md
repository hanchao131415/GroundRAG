### 模块 9a: `app/auth.py`（115 行）— JWT 认证

```
📖 阅读顺序: create_access_token → verify_token → get_current_user
⏱  阅读时间: 10 分钟
```

**带着这些问题读:**

1. JWT 为什么叫"无状态"认证?服务器需要存 session 吗?和无 session 的方案比,这个设计省了什么?
2. `verify_token` 的三重校验分别防什么攻击?(签名、过期时间、sub 字段)如果只校验签名会怎样?
3. `get_current_user` 作为 FastAPI 的 `Depends` 依赖项,是怎么从 `Authorization: Bearer xxx` 头里取出 token 并解码出 user_id 的?它返回什么?
4. `/health` 端点为什么不需要认证?如果它也加认证,会带来什么麻烦?

**精读重点:**
- `create_access_token` — payload 结构(`sub` 存 user_id、`exp` 过期时间、`iat` 签发时间),HS256 对称密钥签名
- `verify_token` — **三重校验**链:`jwt.decode`(验签名+过期)→ 检查 `exp` → 检查 `sub` 存在且非空,任一失败抛 401
- `get_current_user` — FastAPI `Depends` 机制:从 `Authorization` 头提取 Bearer token → 调 `verify_token` → 返回 user_id。**user_id 来自 JWT 解码,不是请求体**

**⚠️ 关键设计理解:**
- `user_id` 是从 JWT 的 `sub` 字段解码出来的,**不是客户端请求体里的字段**。所以客户端伪造 `user_id=admin` 完全无效——服务器只认 token 里签名过的那个 user_id。
- 想改 user_id?必须重签 token,而重签需要 secret。

**费曼自测:**

**精读重点:**

```
📖 阅读顺序: main.py → api.py
⏱  阅读时间: 20 分钟
```

**`main.py` 带着这些问题读:**

1. `initialize()` 按什么顺序初始化各模块?为什么 generation 要先初始化才能做 reranker?
2. `build_knowledge_base()` 里的增量索引和缓存清理是怎么联动的?
3. `ask()` 方法的完整流程是什么?哪几步是"可短路"的?哪几步是"必须走"的?
4. `demo_rbac()` 演示了什么?4 个场景分别验证了什么?

**`app/api.py` 带着这些问题读:**

1. FastAPI 的 `lifespan` 机制和 `get_rag()` 懒加载有什么关系?
2. 5 个端点(login/health/chat/search/stats)分别干什么?其中哪些需要 JWT 认证,哪些不需要?为什么 `/health` 和 `/login` 是开放的?
3. `/api/v1/chat` 的流式分支和非流式分支分别怎么处理?
4. `@limiter.limit("20/minute")` 是怎么生效的?超过限制会返回什么?
5. `get_current_user` 依赖注入是怎么保护 `/chat`、`/search`、`/stats` 这些端点的?如果 token 过期或缺失,请求会发生什么?

**精读重点:**
- `ask()` — **整个系统的编排入口**:缓存查询→短路意图→短路改写→权限检索→生成→存缓存
- `build_knowledge_base()` — 增量索引结果 → 有变化清缓存(坑 29)
- `get_rag()` — 懒加载(首次请求才初始化,后续复用)

**可跳过:**交互式登录的 UI 代码
