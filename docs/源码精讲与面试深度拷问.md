## 〇、一句话定位（面试开场必背）

> "这是一个**企业内部知识库 RAG 问答系统**。一句话讲就是：**带 RBAC 权限隔离的多路混合检索 + cross-encoder 精排 + 多级缓存 + 全链路可观测的 RAG**。
> 它解决的核心问题不是"能不能答"，而是"**答得准、答得安全、答得快、答得可追溯**"——这四点恰好对应检索质量、权限、延迟、可观测四个生产维度。"

**30 秒架构骨架（背下来，画给面试官）：**

```
文档(PDF/DOCX/MD) → 表格感知切分 → bge embedding → FAISS(IP=cosine) + BM25(jieba)
                                                                      ↓
问 → ①缓存(精确/语义) → ②短路路由 → ③查询改写 → ④RBAC先过滤 → ⑤向量+BM25双路召回
   → ⑥RRF融合 → ⑦bge-reranker精排+阈值 → ⑧防幻觉Prompt生成+引用溯源 → ⑨存缓存
                                                                      ↓
全程 Tracer(JSONL) + Langfuse 云端 trace + FallbackLLM 降级链
```

## 一、逐模块源码精讲：是什么 / 为什么这么设计 / 之前怎么做的

### 1. 数据准备 `data_preparation.py` —— 文档怎么进来

**是什么**：多格式加载（PDF/DOCX/XLSX/MD/TXT）→ 元数据增强（source/department/page）→ 表格感知切分 → 父子分块。

**为什么这么设计**：
- **元数据从路径推断部门**（`_enhance_metadata` 取 `parts[-2]`）：约定 `data/docs/{部门}/{文件}`，把"目录结构"当权限标签。这是把"文件组织方式"复用为"权限边界"，零额外标注成本。
- **表格感知切分**（`_split_text_and_tables`）：识别连续的 `|...|` 行，整表当一个 chunk 不切断。因为 `RecursiveCharacterTextSplitter` 按字数切，会把一张 6 行工资表切散，丢掉 P4/M2 行。
- **Excel 生产级加载**（2026-07 新增）：pandas + openpyxl，8 项加固——空 Sheet 四重检测、合并单元格 ffill、列名清洗（换行/空格/重复去重）、转置表自动检测（GPU 对比表：型号为列→自动转置为行）、**列摘要块**（关键列值聚合，搜"Blackwell"直接命中摘要而非逐行扫）。
- **OCR 兜底**：PDF 扫描件 `page.get_text()` 返回空时，pytesseract 300dpi 中英文 OCR，依赖未安装时优雅降级跳过。
- **父子分块**：小块（精准检索）→ 回溯父文档（完整上下文）。
- **`content_hash`**：每个 chunk 算 md5，给增量索引用。

**之前怎么做的（演进）**：
| 阶段 | 做法 | 问题 |
|---|---|---|
| demo | 只支持 markdown，`MarkdownHeaderTextSplitter` 按 `#` 切 | PDF 无 `#` 结构，切不了 |
| v1 | 加 `RecursiveCharacterTextSplitter` 通用切 | 表格被切散（坑22） |
| 现在 | 表格感知 + Excel 生产级 + OCR 兜底 + 列摘要 | 6 格式全覆盖 |

**面试官会追问的点（见第六节）**：表格识别用"`|` 开头"的启发式，真实 PDF 表格提取出来不一定是 `|` 分隔的，会漏识别。

### 2. 索引构建 `index_construction.py` —— 向量化与存储

**是什么**：bge-small-zh embedding（`normalize_embeddings=True`）→ FAISS `IndexFlatIP`（内积=cosine）→ 增量重建（content_hash 对比）。

**为什么用 IP 不用 L2（坑19，高频题）**：
- bge 向量归一化后，**内积 IP = cosine 相似度**（数学等价）。
- FAISS 默认 `IndexFlatL2` 返回的是欧氏距离，范围 0~2，**越小越好**；cosine 范围 0~1，**越大越好**。两者方向相反，搞反就全错。
- 归一化下 `L2 = √(2-2·cos)`，单调对应、排序等价，但 IP 数值直观（0~1 相似度）、阈值好定、性能略快。
- 代码里先 `FAISS.from_documents`（默认 L2）建库拿向量，再 `reconstruct_n` 取出向量，换成 `IndexFlatIP` 重新 add。**为什么不直接建 IP？** 因为 LangChain 的 `FAISS.from_documents` 内部写死了 L2，只能建完再偷梁换柱。

**增量索引（坑27）+ P0 僵尸向量修复**：每个 chunk 有 `content_hash`，重建时对比 `chunk_hashes.json`。**检测到任何变化时全量重建索引**（而非增量 add_documents），因为 `IndexFlatIP` 不支持删除向量——增量追加会让旧版本文件的僵尸向量永远残留，导致检索返回重复内容（坑33，P0 修复）。

**之前怎么做的**：每次全量重建。文档一改，全部重新 embedding，慢且浪费。

### 3. 检索 `retrieval_optimization.py` —— 系统的心脏 ⭐⭐⭐⭐⭐

**`hybrid_search` 四阶段**：
1. **宽松召回**：向量 `similarity_search_with_score(k=top_k*2)` + cosine≥0.3 阈值过滤；BM25 召回（中文 jieba 分词 + 70+ 英文技术词注册）。
2. **RRF 融合**：`score = Σ 1/(k+rank)`，k=60 平滑，用 `chunk_id` 去重。
3. **MMR 多样性去重**（P1 修复）：先 `content_hash` 精确去重，再 3-gram Jaccard MMR（`score=0.7×relevance−0.3×max_similarity`），候选 >20 时只 Top-15 做 MMR（性能保护）。覆盖全部 3 个检索路径。
4. **Reranker 精排**（可选，Windows CPU 下已禁用）：bge-reranker cross-encoder，失败时 try-except 降级跳过。

**核心设计哲学**：bi-encoder 宽松召回保不漏 → RRF 融合 → MMR 去重（防 PDF 页眉重复浪费 reranker 算力）→ cross-encoder 精排保高精度。

**A/B 实验硬证据**：负例拒答率 0%→100%，综合准确率 0%→97.3%。代价正例召回 100%→97.3%——企业场景答错比漏答危险。

**中文 BM25 + 技术词典**：模块加载时注册 70+ 企业技术英文词（Redis/Docker/InfiniBand/H100/FP16…），防 jieba 把"Redis"切成 Re+dis（坑34）。GPU/IDC 术语来自实际文档提取。

**BM25 持久化**（P1 优化）：pickle 保存/加载 `bm25_index.pkl`，10 万级 chunk 启动从"数秒"→"毫秒"。加载失败自动回退重建。

**权限检索 `permission_aware_search`**：admin（`*`）全库 `hybrid_search`；普通用户走 `_rbac_subindex_search`（部门子索引真·先过滤，从全库切片零重复嵌入）或 `metadata_filtered_search`（回退路径）。

> ✅ **#A1 已部分修复（2026-07）**：`_rbac_subindex_search` 使用部门子索引做真·先过滤。

### 4. 生成 `generation_integration.py` —— 防幻觉与溯源

**防幻觉 Prompt 四条约束**：
1. 只能依据上下文 → 防"参数知识注入"（LLM 用自己知识答）。
2. 无依据说"未找到" → 防硬答（负例拒答的最后一道闸）。
3. 每个事实标 `[文档X]` → 可溯源验证。
4. 不答无关内容 → 防跑题。

**`_build_context`**：拼接检索结果为 `【文档1】来源/页/部门\n内容`，`max_length=3000` 截断防 Lost-in-the-Middle。

**查询改写 / 意图路由**：模糊问题才走 LLM 改写/路由（`is_simple_query` 短路）。

**之前怎么做的**：Moonshot 写死 + 烹饪 prompt，一次 invoke。现在 OpenAI 兼容协议 + 防幻觉约束 + LCEL chain + 异步。

### 5. 缓存 `cache_service.py` —— 多级缓存与误命中防御

**两级**：精确（query+权限签名 md5，O(1) 哈希索引）+ 语义（cosine≥0.85，同 perm_sig 内搜索）。

**四重误命中校验**（`_safe_to_cache_hit`，这是整个缓存设计的精华）：
| 校验 | 拦截的场景 |
|---|---|
| 长度差>40% | "p2工资" vs "p1和p2相差多少"（坑23） |
| 实体不匹配 | 新 query 有 P1，缓存只有 P2 |
| 比较意图不一致 | "差多少" vs "多少"（坑23） |
| **数字必须完全一致** | "满1年" vs "满3年"（sim 0.94 但答案不同，坑25） |

**权限签名隔离（坑26）**：`_hash(query, perm_sig)`，HR 和财务问同一问题各自缓存。否则缓存层绕过 RBAC = 后门。

**O(1) 精确查找**：维护 `_hash_idx: Dict[str, int]`（哈希→列表下标），精确匹配不随缓存增长变慢。

**短路路由 `is_simple_query`**：明确问题（≥4字、非纯口语）跳过 LLM 改写/路由，省 6~20 秒。

### 6. 认证与权限 `app/auth.py` + `user_service.py`

**三层安全模型**：① 认证（JWT 验签拿 user_id）→ ② 授权（user_id→部门）→ ③ 过滤（部门→检索过滤）。

**JWT 三重校验**：签名（防伪造）+ 过期（防盗用）+ sub 非空（防空 payload）。

**关键**：user_id 从 token 解码，**不是请求体**。客户端伪造 `{"user_id":"admin"}` 无效（坑30）。

> ⚠️ **但这里有现有文档回避的真问题（见"找茬 #3"）**：`/auth/login` **不要密码**，只要 user_id 存在就发 token。所以这其实是"身份识别"不是"身份认证"——知道 admin 这个 user_id 就能拿 admin token。严格意义上 RBAC 的第一环是漏的。

---

### 7. LLM 工厂 + 降级链 `llm_factory.py` + `llm_fallback.py`

**工厂（注册表模式）**：`PRESETS`（供应商预设）+ `_REGISTRY`（协议适配器，装饰器注册）。新增供应商加一条预设 + 一个适配器，不改老代码（开闭原则）。

**降级链（组合模式）**：`FallbackLLM` 持有 LLM 列表，invoke 依次尝试，带**熔断冷却**（失败 provider 60s 不重试）+ **成功记忆**（`_last_success_idx`，粘在最近成功的 provider，避免抖动）。duck-type 兼容 LangChain Runnable。

### 8. 可观测 `tracer.py` + `observability.py`

自研轻量 trace（JSONL）：每步耗时/输入输出/token/cost。Langfuse 云端做可视化，降级设计（key 没配返回 None 不影响主流程）。trace 是性能优化的依据（坑20/24 靠 trace 定位改写占 80% 延迟）。

## 二、难点与挑战（深度）

| 难点 | 本质 | 本项目的解法 |
|---|---|---|
| **精度/召回权衡** | 阈值设松收负例（幻觉），设紧漏正例 | 两段式：宽松召回保召回 + rerank 精排保精度 |
| **向量距离重叠** | bi-encoder 分不清"语义近但无关" | cross-encoder rerank（坑10，A/B 证明） |
| **语义缓存的根本矛盾** | 语义相似 ≠ 答案相同 | 四重校验 + 权限签名（宁可漏命中不误命中） |
| **缓存一致性** | 索引更新了缓存还是旧的 | 增量索引返回 has_changes → 自动清缓存 |
| **权限与检索耦合** | 先检索后过滤会召回不足 | （声明）先过滤后检索 + 缓存权限隔离 |
| **中文 BM25 失效** | 默认空格分词，整句当一个词 | jieba 分词 |
| **LLM 单点** | provider 挂了全站不可用 | FallbackLLM 降级链 + 熔断 |
| **复杂文档表格** | 递归切分散表格 | 表格感知切分 |
| **能力边界** | 多跳/计算/比较 RAG 做不了 | 老实拒答，明确这是 Agent 的活（坑17） |

### 找茬 #A2：API 层 `current_user` 并发竞态 ⚠️⚠️ ✅ 已修复

**代码事实**（`app/api.py:81-90, 137-141`）：
```python
_rag: EnterpriseRAGSystem = None   # 进程级单例
def get_rag(): ...                  # 所有请求共享同一个 _rag

@app.post("/api/v1/chat")
async def chat(..., user_id = Depends(get_current_user)):
    rag = get_rag()
    rag.current_user = user_id      # ← 往单例上写共享可变状态！
    answer = await asyncio.to_thread(rag.ask, ...)  # ← ask 内部读 self.current_user
```

**后果**：两个用户并发请求 → A 写 `current_user="zhangsan"`，B 写 `current_user="lisi"`，A 的 `ask`（在另一个线程里）可能读到 `"lisi"` → **用错部门做权限过滤 → 跨权限泄露或搜不到**。GIL 救不了，因为写和读跨了 `await`/线程切换。

**根因**：`EnterpriseRAGSystem` 是为单用户 CLI 设计的（`current_user` 是实例状态），API 直接复用了它，把"请求级状态"塞进了"单例级状态"。

**诚实的回答**：
> "这是真 bug。`current_user` 不该挂在单例上。正确做法是**请求级上下文**：把 user_id/departments 作为参数显式传进 `ask(question, user_departments=...)`（方法签名其实已经支持了！），API 层从 JWT 解出 user_id → 查 departments → 显式传参，绝不写单例。我 CLI 那条路是对的，API 这条路图省事复用了，是遗留债。"

> 注：`ask()` 签名是 `ask(self, question, stream=True, user_departments=None)`，**已经支持传参**。`demo_rbac` 就是传参用的。但 `chat` 端点偏走了 `rag.current_user = user_id` 这条共享状态的路。修复成本极低——这是"知道哪错了"就能修的典型。

> ✅ **已修复（2026-07）**：

### 找茬 #A3：`/auth/login` 不要密码，"认证"名不副实 ⚠️（部分修复：已加限流）

**代码事实**（`app/api.py:111-132` + `user_service.py:66`）：
```python
def login(req: LoginRequest):           # LoginRequest 只有 user_id，没有 password
    user = rag.user_service.authenticate(req.user_id)  # authenticate 只查 user_id 存不存在
    token = create_access_token(req.user_id, ...)      # 存在就发 admin token
```

**后果**：任何知道 `admin` 这个 user_id 的人，`POST /auth/login {"user_id":"admin"}` 直接拿到管理员 token。**这是"身份识别"（identification）不是"身份认证"（authentication）**——没有任何只有本人知道的凭证。JWT 验签只能证明"token 是我发的"，不能证明"拿 token 的人是本人"。

**面试官会怎么问**：「你的 RBAC 三层里'认证'那层，login 不要密码，知道 user_id 就发 token，这算认证吗？」

**诚实的回答**：
> "不算，您说得对。这是 demo 妥协——为了不引入密码哈希/SSO 的复杂度，我把'认证'降级成了'识别'。生产里这层必须补：要么密码（bcrypt 哈希）+ 登录限流防爆破，要么接企业 SSO/OAuth。JWT 解决的是'发出去的 token 不可伪造'，不解决'发给谁'——后者要靠登录时的凭证校验。我的 `/auth/login` 现在连限流都没单独配（只有全局 30/min），爆破 user_id 列表理论上可行。"

> 🔶 **部分修复（2026-07）**：`/auth/login` 已加 `@limiter.limit("5/minute")` 防 user_id 枚举爆破（slowapi 已引入，零成本）。**密码/SSO 仍未补**——这属"生产必做"范畴，留给真实接入企业身份系统时实现（bcrypt 密码哈希 或 OAuth/OIDC）。面试讲法："我先把无成本的反爆破补上了；真上生产这层必须接 SSO，不是自己撸密码。"

### 找茬 #A4：索引增量更新有僵尸向量 + 删除未处理 🔴 P0 已修 / 🟡 删除待做

**代码事实**（`index_construction.py:build_incremental`）：原实现 `existing.add_documents(new_chunks)` 只追加不删除，编辑文件后旧版本 chunk 的向量永远残留（IndexFlatIP 无 remove API）。删除文件同理——孤儿向量永不清理。

**P0 修复（2026-07）**：检测到任何变化时**全量重建索引**（`self.build_vector_index(chunks)`），而非增量追加。代价是每次文档变化都要重新 embedding 所有 chunk，但对中小规模知识库可接受。

**删除仍未解决**：如果文件被物理删除（从 data/docs/ 移除），下一次 `load_documents` 不再加载它 → chunks 不含它的 chunk → `build_incremental` 检测到旧 chunk_key 不在新 chunks 中 → 标记为 new → 触发全量重建。所以**删除实际上通过"全量重建"间接解决了**——新索引不包含已删除文件的向量。

> ✅ **已修复（2026-07，P0）**：全量重建消除僵尸向量。面试讲法："IndexFlatIP 不支持 remove，我选择检测到变化时全量重建——牺牲 embedding 时间换正确性。对中小知识库（<1000 chunks），embedding 只需十几秒，可接受。"

### 找茬 #A6：FAISS `allow_dangerous_deserialization=True` 的 RCE 风险

**代码事实**（`index_construction.py:189-193`）：
```python
self.vectorstore = FAISS.load_local(self.index_save_path, self.embeddings,
                                    allow_dangerous_deserialization=True)
```
FAISS 的 `save_local`/`load_local` 用 pickle。`allow_dangerous_deserialization=True` 意味着**任何人能写 `index.pkl` 就能远程代码执行**。

**后果**：若 `index_save_path` 可被攻击者写入（如共享存储、容器挂载），构造恶意 pickle 即 RCE。

**诚实的回答**：
> "这是已知的安全代价。LangChain 默认禁止反序列化就是为了防这个，我开了 `allow_dangerous_deserialization=True` 是因为要本地持久化索引。生产缓解：① 索引目录严格权限控制（只读、不挂载外部）；② 改用 FAISS 原生 `write_index`/`read_index`（存纯向量，不 pickle Python 对象）；③ 或换 Milvus（gRPC 协议，无 pickle）。"

### 找茬 #A7：缓存只增不删，无淘汰策略

**代码事实**（`cache_service.py:176-191`）：`put` 每次追加写 `qa_cache.jsonl`，`_cache` 列表只增。只有 `clear()`（全量清，索引更新时触发）能减。**没有 LRU、没有 TTL、没有容量上限**。

**后果**：长期运行 `qa_cache.jsonl` 无限增长；`get` 的语义匹配是**O(N) 线性扫**（`for item in self._cache`），缓存越大查询越慢——缓存本身会成为瓶颈。

**诚实的回答**：
> "缓存现在是 append-only + 线性扫描，数据量大了会退化。生产方案：① 上 Redis（自带 LRU/TTL/原子操作）；② 内存里维护向量索引（FAISS）加速语义匹配，而非线性扫；③ 加容量上限 + LRU 淘汰。现在这个实现是'验证缓存思路'的 MVP，不是生产存储。"

### 找茬 #A8：`get_current_user` 的 HTTPBearer 依赖注入失效 ⚠️⚠️ ✅ 已修复

> 这条是**修复 A2 写回归测试时意外挖出来的**——写 TestClient 集成测试才发现，否则永远不会暴露。典型的"没有集成测试 = 永远不知道端点是坏的"。

**代码事实**（`app/auth.py` 原始写法）：
```python
security = HTTPBearer(auto_error=False)

async def get_current_user(
    request: Request,
    credentials: Optional[HTTPAuthorizationCredentials] = security,  # ← 没 Depends()！
) -> str:
    if credentials is None or not credentials.credentials:  # ← 这里炸
```

**症状**：任何带 `Depends(get_current_user)` 的端点（`/chat`、`/search`、`/stats`）被真实 HTTP 调用时，抛 `AttributeError: 'HTTPBearer' object has no attribute 'credentials'`。

**根因**：`credentials = security` 把 `HTTPBearer` 实例**直接当默认值**传进来了——FastAPI 没把它识别成依赖项去调用 `HTTPBearer.__call__` 解析凭证，于是 `credentials` 拿到的是 `security` 对象本身（HTTPBearer），而非 `HTTPAuthorizationCredentials`。访问 `.credentials` 自然 AttributeError。

**为什么一直没发现**：现有 `TestJWTAuth` 只测了 `create_access_token` / `verify_token` 这两个纯函数，**从没通过 HTTP 调用受保护端点**。单元测试全绿，端点实际是死的。这是"单元测试覆盖率 100% 但集成层面是 0"的经典反例。

**修复**：显式包 `Depends`——
```python
from fastapi import Depends
async def get_current_user(
    request: Request,
    credentials: Optional[HTTPAuthorizationCredentials] = Depends(security),
) -> str:
```

**验证**：修复后诊断脚本 `/chat` 返回 200，`ask` 收到 `user_id=zhangsan, depts=['HR']`。

**💡 面试话术**（这条极加分，证明你重视集成测试）：
> "我写 A2 的回归测试时用 TestClient 打 `/chat`，发现端点根本走不通——HTTPBearer 的依赖注入写错了，`credentials` 拿到的是 security 对象本身。但单元测试一直全绿，因为只测了 token 的纯函数，没测端点集成。这让我意识到**单元测试覆盖率不等于功能正确性**——关键链路必须有集成测试（哪怕用 mock 跑通 HTTP 层）。修复后我专门加了 3 条 TestClient 测试锁住这条端到端路径。"

## 七、面试深度拷问 Q&A（30 题，严格面试官版）

> 比现有 `面试攻防手册.md` 更硬。每题给：**意图 / 标答要点 / 代码证据 / 陷阱（错误回答长啥样）**。
> 带 ⚠️ 的是"杀手锏题"，答不上来基本掉档。

### A. 检索内核

**Q1. 你的检索是几路融合？为什么不全用向量？**
- 意图：考检索基本功。
- 标答：向量（语义）+ BM25（关键词）两路，RRF 融合。向量不懂"字面精确词"，BM25 不懂"语义变形"，互补（坑6）。RRF 用排名信息 `1/(k+rank)`，比简单取并集好。
- 陷阱：「全用向量就行」——错，BM25 抓精确术语（产品型号、人名）向量抓不到。

**Q2. RRF 为什么用 `1/(k+rank)`，k=60 怎么定的？**
- 标答：排名越靠前权重越高。k 是平滑常数，控制头部和尾部排名的分数差；k=60 是论文经验值（原始 RRF 论文），让第 1 名和第 2 名的差距不至于过大。可调，但 60 是公认稳健默认。
- 陷阱：「随便选的」——要有依据。

**Q3. ⚠️ 你说权限是先过滤后检索，但 `metadata_filtered_search` 里向量检索器是全库 `as_retriever`，这不是先检索后过滤吗？**
- 见**第五节 #A1**。这是杀手锏题。能主动承认 + 给出 Milvus/反向索引的解法 = 加大分；硬撑"是先过滤"= 直接掉档（面试官读过代码）。

**Q4. reranker 为什么是 cross-encoder，向量检索是 bi-encoder？为什么两段式？**
- 标答：bi-encoder query/doc 分别编码（快、可离线建库、糙）；cross-encoder 拼一起编码（慢、不能建库、准）。两段式：bi 保召回，cross 保精度。A/B 证明负例拒答 0%→100%。
- 陷阱：「reranker 就是为了更准」——要说清 why 两段式（cross 慢，不能对全库跑）。

**Q5. rerank 牺牲了 2.7% 正例召回，值得吗？**
- 标答：值得。企业场景答错（幻觉）比漏答危险。97.3% 召回 + 100% 拒答，优于 100% 召回 + 0% 拒答。这是显式的精度/召回权衡。

**Q6. cosine 和 L2 在归一化向量下什么关系？你用哪个？**
- 标答：`L2=√(2-2cos)`，单调对应、排序等价，但方向相反（L2 越小越好，cos 越大越好）。我用 IP（归一化下=cosine），数值直观、阈值好定（坑19）。

### B. 缓存与性能

**Q7. ⚠️ "满1年年假" 和 "满3年年假" 语义相似度 0.94，你缓存怎么不答错？**
- 标答：四重校验里的"数字必须完全一致"拦住了（坑25）。`re.findall(r'\d+')` 提取数字集合比较，{1}≠{3} → 否决。体现"语义相似≠答案相同"。
- 陷阱：只说"阈值调高"——阈值调多高都会误伤合法改写，必须靠结构化校验。

**Q8. 缓存怎么防跨权限泄露？**
- 标答：权限签名嵌入 hash key（坑26）。HR 和财务问同一问题，key 不同，各自缓存；语义匹配也只在同 `perm_sig` 内搜。

**Q9. ⚠️ 你的 trace 发现改写占 80% 延迟，怎么优化的？优化后还有什么问题？**
- 标答：短路路由——明确问题（`is_simple_query`）跳过 LLM 改写/路由，6355ms→0ms（坑20/24）。
- 陷阱（主动讲）：`is_simple_query` 是粗糙启发式（`len<4` 判模糊），中文短问题可能误判；且大部分问题都走短路，改写几乎不触发，等于"优化掉了改写"而非"智能选择改写"。更彻底的方案是规则 + 小模型分级。

**Q10. 缓存击穿/雪崩怎么防？**
- 标答：诚实说现在没防（单机小规模）。生产方案：热点 key 加锁单飞（singleflight）、TTL 加随机抖动防雪崩、熔断兜底。

### C. 安全与权限

**Q11. ⚠️ 客户端 POST `user_id=admin` 能拿到管理员权限吗？**
- 标答：不能。user_id 从 JWT 解码，不是请求体（坑30）。但**主动补刀**：我的 `/auth/login` 不要密码，知道 admin 这个 user_id 就能登录拿 token（#A3），所以"认证"层其实是漏的，生产要补密码/SSO。这种主动暴露弱点反而加分。

**Q12. JWT 为什么无状态？过期了怎么办？**
- 标答：服务端不存 session，水平扩容不需共享存储。过期 `exp` 由 `jwt.decode` 自动校验抛 401，客户端重登。陷阱：没 refresh token，长期会话体验差——主动提可加 refresh。

**Q13. JWT 密钥泄露怎么办？**
- 标答：轮换密钥（支持多密钥并存期）+ 强制所有 token 重新签发。HS256 单密钥的弱点——多服务该上 RS256（公私钥分离）。

**Q14. ⚠️ 你的 RBAC 是"先过滤后检索"，但 FAISS 不支持元数据预过滤，你到底怎么实现的？**
- 同 Q3/#A1。诚实讲妥协。

**Q15. ⚠️ 多个用户同时请求 `/chat`，你的系统安全吗？**
- 见 **#A2**。`current_user` 单例竞态，可能跨权限。这是并发安全杀手锏题。

### D. 可靠性与工程化

**Q16. LLM 挂了怎么办？**
- 标答：FallbackLLM 降级链，primary→fallback，熔断冷却 60s，成功记忆防抖动（坑31）。组合模式 duck-type 兼容。

**Q17. ⚠️ stream 模式 LLM 中途挂了，能切备用吗？**
- 标答：不能优雅切。已经吐了一半 token，备用 provider 不知道前文，只能整段重来或报错（`llm_fallback.py` 的 stream 注释有讲）。这是流式降级的本质难点。

**Q18. 熔断和健康检查区别？你的冷却算哪个？**
- 标答：健康检查是主动探测；我的冷却是"被动熔断"——失败一次后冷却期内不试，是客户端版断路器。区别：我不主动探测，靠实际请求的失败信号。

**Q19. 增量索引怎么判断"变了"？能处理删除吗？**
- 标答：content_hash(md5) 对比。**删除处理不了**（#A4），FAISS flat 无高效 remove，孤儿向量残留。主动讲这是缺口。

**Q20. 文档更新了，缓存怎么办？**
- 标答：增量索引返回 `has_changes`，main.py 自动 `cache.clear()` 全量清（坑29）。权衡：全量清而非精准失效，因缓存小、文档更新不频。追问"10万条缓存呢"→ 答 Redis + 反向索引(doc_id→cache_keys) 精准失效。

**Q21. ⚠️ FAISS `allow_dangerous_deserialization=True` 你知道风险吗？**
- 见 **#A6**。pickle RCE。

### E. 评测与数据

**Q22. 评测三维度怎么算？为什么不用 ragas？**
- 标答：context_recall/faithfulness/answer_relevancy，LLM-as-Judge 打 0/1（`run_eval.py`）。不用 ragas 因依赖重、3.14 编译失败、黑箱；自研能讲清每个指标。

**Q23. ⚠️ 你的评测集自编，裁判和生成用同一个 LLM，可信吗？**
- 标答：诚实承认有自偏（作者偏见 + LLM 自偏好）。缓解：fuzzy_eval 用 LLM 生成变体；裁判换 provider；找同事盲评；接真实日志。这是评测严谨性杀手锏题——硬说"可信"就掉档。

**Q24. 评测集 41 条够吗？怎么扩到无偏？**
- 标答：不够。扩方向：真实用户 query 日志、多标注者、bad case 归类、分层抽样（事实型/比较型/拒答型）。

### F. 架构与扩展

**Q25. 能扛多少 QPS？做过压测吗？**
- 标答：IndexFlatIP 暴搜 O(N)，10万级可接受，百万级换 IVF/HNSW。有 locustfile.py。诚实说数据量小（几十文档），压测主要验证链路，没到瓶颈。

**Q26. 怎么从单机扩到分布式？**
- 标答：单机（FAISS/缓存/JSONL 都进程内）。演进：缓存→Redis → FAISS→Milvus → API 无状态多副本。JWT 无状态已为扩容铺路。

**Q27. ⚠️ 多副本部署时，你的 `current_user`/缓存/FAISS 哪些会出问题？**
- 标答：current_user 单例竞态（#A2）每副本内部就有；缓存进程内不共享（要 Redis）；FAISS 索引每副本各加载一份（内存浪费，要分布式向量库）。这题综合 #A2/#A7。

**Q28. 为什么用 LangChain？它的坑？**
- 标答：快速搭 LCEL chain。坑：版本连环升级（锁版本）、流式拿不到 usage_metadata（坑18）、FAISS 接口写死 L2。生产可考虑去掉 LangChain 直接调 SDK，减少抽象层。

### G. 认知深度

**Q29. RAG 的能力边界在哪？什么时候该上 Agent？**
- 标答：RAG 擅长事实查询，不擅长多跳/计算/比较（坑17："李四年假比张三多多少"答不出）。需要多步推理+工具调用时上 Agent。**知道边界比堆功能重要。**

**Q30. ⚠️ 如果让你重做这个项目，你会改什么？**（终极题，考反思深度）
- 标答框架（按优先级）：
  1. **API 层去掉单例共享状态**（#A2），请求级上下文显式传参——这是 correctness 问题，最高优先级。
  2. **向量库换支持元数据过滤的**（Milvus/Qdrant，#A1/#A4/#A6 一起解）。
  3. **缓存上 Redis**（#A7 + 并发安全）。
  4. **认证补密码/SSO + 登录限流**（#A3）。
  5. **评测解自偏**（#23）。
  - 这题答得好 = 你真懂这个系统的优缺点，而不是只会吹。
