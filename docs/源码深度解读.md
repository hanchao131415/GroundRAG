## 一、工厂模式与注册表:`llm_factory.py`

### 先看整体设计

```
用户 .env 写 LLM_PROVIDER=deepseek
  → create_llm("deepseek", ...)
  → 查 PRESETS["deepseek"] → {"protocol":"openai", "base_url":"...", "model":"deepseek-chat"}
  → protocol="openai" → 查 _REGISTRY["openai"] → _make_openai()
  → 返回 ChatOpenAI 实例
```

**核心问题:为什么不用 if-else,要上注册表?**

如果这样写(常见新手写法):
```python
if provider == "deepseek":
    llm = ChatOpenAI(...)
elif provider == "zai":
    llm = ChatAnthropic(...)
elif provider == "gemini":
    llm = ChatGoogleGenerativeAI(...)
```

每加一个新供应商,就得改这个函数——"对修改开放"(违反开闭原则)。注册表把"适配器函数"和"工厂逻辑"解耦了。

### 注册表怎么实现的(装饰器模式)

```python
_REGISTRY: Dict[str, AdapterFactory] = {}

def register_adapter(protocol: str):
    """装饰器:注册一个协议适配器"""
    def decorator(func):
        _REGISTRY[protocol] = func
        return func
    return decorator

@register_adapter("openai")
def _make_openai(api_key, base_url, model, ...):
    return ChatOpenAI(base_url=base_url, ...)
```

**`@register_adapter("openai")` 在函数定义时就执行了**——把 `_make_openai` 存入 `_REGISTRY["openai"]`。所以 Python 解释器加载这个文件时,注册表就自动建好了。不是运行时注册的,是**导入时注册**。

**为什么选装饰器而不是手动注册?**

手动注册:
```python
_REGISTRY["openai"] = _make_openai
_REGISTRY["anthropic"] = _make_anthropic
```
也能用。但装饰器的好处:**适配器函数定义和注册在同一处**,不会忘了注册,也不怕改名后漏改注册表。一眼看出"这个函数是一个适配器"。

### 预设怎么工作

```python
PRESETS = {
    "deepseek": {"protocol": "openai", "base_url": "https://api.deepseek.com/v1", "model": "deepseek-chat"},
    "zai": {"protocol": "anthropic", "base_url": "https://api.z.ai/api/anthropic", "model": "glm-4.6"},
}
```

用户 `.env` 写 `LLM_PROVIDER=deepseek`,工厂去 `PRESETS` 查——**用户不需要知道"什么协议""什么 base_url"**,只需知道供应商名。这就把"配置复杂度"封装在预设里。

### 优先级:显式参数 vs 预设

```python
def create_llm(provider, api_key, base_url="", model="", ...):
    preset = PRESETS.get(provider.lower(), {})
    final_base_url = base_url or preset.get("base_url", "")
    final_model = model or preset.get("model", "")
```

**`base_url or preset.get(...)`:**Python 的 `or` 短路——显式传入的 `base_url` 非空,就直接用;为空才兜底预设。所以用户 `.env` 写 `LLM_BASE_URL=...` 会覆盖预设,**不写就用预设**。

但这里有一个陷阱:`or` 不会区分"空字符串"(没传)和"用户传了空值",两者都走预设兜底。好在 `.env` 不会出现空值的 base_url,所以安全。

## 二、检索的"心脏":`retrieval_optimization.py`

### hybrid_search:两阶段检索

整个系统的核心函数,约 50 行。我把它拆解成"为什么要这样设计"。

**阶段1:宽松召回**

```python
raw = self.vectorstore.similarity_search_with_score(query, k=top_k * 2)
vector_docs = []
for doc, sim in raw:
    doc.metadata["vector_sim"] = round(float(sim), 4)
    vector_docs.append(doc)
vector_docs = [d for d in vector_docs if d.metadata["vector_sim"] >= score_threshold]
```

**`k=top_k * 2`**:召回 2 倍候选。为什么?因为后面有 RRF 融合+rerank 精排,宽松多召回保证"不漏",精排再"去噪"。如果 k=top_k,候选太少,精排没得选。

**`vector_sim >= score_threshold`**(默认 0.3):阈值过滤。不加这个,topK 会硬返回 K 个,即使余弦相似度只有 0.1 也返回——负例无法拒答(坑 8/9)。

**阶段2:RRF 融合**

```python
candidates = self._rrf_rerank(vector_docs, bm25_docs)
```

向量和 BM25 各自召回,RRF 融合去重。为什么不用简单取并集?因为 RRF 考虑了**排名信息**——向量排第 1 的文档,比 BM25 排第 5 的更重要,融合时给它更高权重。`1/(k+rank)` 就是这个权重的体现。k=60 平滑排名差异。

**阶段3:Reranker 精排**

```python
if self.reranker:
    candidates = self.reranker.rerank(query, candidates)
    kept = [d for d in candidates if d.metadata.get("rerank_score", 0) >= rerank_threshold]
    final = kept[:top_k]
    return final
```

**为什么 RRF 之后还要 Reranker?** RRF 只是"排名融合",不是"相关性判断"。向量检索可能把"语义相近但实际无关"的文档排前面(坑 10:距离重叠)。Reranker(cross-encoder)重新精确判断每对(query,doc)的相关性,去噪+重排。

**`rerank_threshold`**(默认 0.3):rerank 后再做一次阈值过滤——负例(无关文档)在 rerank 阶段得分很低(0.0001),被阈值直接丢弃,实现拒答。

### `_rrf_rerank`:为什么用 `chunk_id` 而不是 `md5(content)` 去重

```python
for rank, doc in enumerate(vector_docs):
    doc_id = doc.metadata.get("chunk_id") or hashlib.md5(doc.page_content.encode('utf-8')).hexdigest()
    doc_objects[doc_id] = doc
```

之前用的是 `md5(page_content)`——如果两个不同 chunk 内容恰好相同(比如重复的模板文本),会被错误去重(坑 12:review 指出)。改成 `chunk_id` 优先:`chunk_id` 是 UUID,保证唯一;没有 chunk_id 才降级用 md5。

### `permission_aware_search`:权限怎么和检索结合

```python
def permission_aware_search(self, query, user_departments, top_k=3):
    if "*" in user_departments:
        return self.hybrid_search(query, top_k)  # 管理员:全库

    allowed = list(set(user_departments + ["公共"]))
    return self.metadata_filtered_search(query, {"department": allowed}, top_k)
```

**管理员路径**:`"*"` 跳过全部过滤,直接全库检索。

**普通用户路径**:可见部门列表 + "公共"(公共文档所有人都能看)。然后走 `metadata_filtered_search`,按 `department` 字段过滤。

**关键设计:先过滤后检索(坑 7)**

```python
def metadata_filtered_search(self, query, filters, top_k=5):
    allowed_chunks = [doc for doc in self.chunks if self._match_filters(doc.metadata, filters)]
    # 然后在 allowed_chunks 子集上重建检索器并检索
```

**为什么不是"先检索后过滤"?** 如果先 `hybrid_search(top_k=3)` 召回 3 个,再过滤——这 3 个全不在权限内,过滤后变空。**有权限的用户反而搜不到自己的文档**(坑 7)。先过滤 candidates 再检索,保证不会"召回,但全被过滤掉"。

## 三、多级缓存的防误命中设计:`cache_service.py`

### 两级缓存:精确 + 语义

```python
def get(self, query, departments=None):
    perm_sig = self._perm_signature(departments)  # "HR,公共"
    qh = self._hash(query, perm_sig)              # md5("年假几天|HR,公共")

    # ① 精确匹配(相同 query + 相同权限 = 相同答案)
    for item in self._cache:
        if item["hash"] == qh:
            return {"answer": item["answer"], "cache_type": "exact"}

    # ② 语义匹配(同义词改写,如"密码几天换"≈"密码更换周期")
    # 只在同 perm_sig 的条目里搜,防跨权限泄露
    for item in self._cache:
        if item.get("perm_sig") != perm_sig: continue  # 权限隔离
        sim = cosine(query_vec, item.embedding)
        ...

    if best_sim >= 0.85 and self._safe_to_cache_hit(query, cached_q, best_sim):
        return {"answer": best_item["answer"], "cache_type": "semantic"}
    return None
```

**两级设计背后的原理:**
- 精确匹配:完全相同的问题(含权限),哈希命中,0.1ms。最安全,零误命中风险
- 语义匹配:同义改写(如"密码几天换"≈"密码更换周期"),cosine≥0.85 + 四重校验才命中

**权限签名为什么嵌入 hash?** 不嵌的话:张三(HR)问"年假几天"存缓存,李四(财务)问完全一样的"年假几天",哈希值相同 → 李四拿到 HR 的答案。这等于**缓存后门绕过了 RBAC**(坑 26)。

### 四重校验的逐层含义

```python
def _safe_to_cache_hit(query, cached_q, sim):
    # ① 长度差异 > 40% → 可能不同问题
    # ② 实体不匹配(新query有P1,缓存无P1) → 问的不同对象
    # ③ 比较意图不一致(一个问"差多少",一个问"多少") → 不同类问题
    # ④ 数字不一致("满1年" vs "满3年") → 答案肯定不同
    return True  # 四关全过
```

每层解决一个真实误命中场景:

| 场景 | 触发哪层 |
|---|---|
| "p2工资"命中"p1和p2相差多少"的缓存 → 答非所问 | ①长度 ②实体 ③比较 |
| "工作满1年"命中"工作满3年"的缓存 → 答错 | ④数字 |
| "年假几天"HR的缓存被财务命中 → 权限泄露 | 权限签名(外层) |

## 四、表格感知切分:`data_preparation.py`

### `_split_text_and_tables`:表格怎么被识别并保护

```python
def _split_text_and_tables(self, content):
    blocks = []
    lines = content.split("\n")
    i = 0
    while i < len(lines):
        line = lines[i]
        if "|" in line and line.strip().startswith("|"):
            # 向后收集连续的表格行
            table_lines = [line]
            j = i + 1
            while j < len(lines) and lines[j].strip().startswith("|"):
                table_lines.append(lines[j])
                j += 1
            flush_text()  # 先把积累的文本写入 text 块
            blocks.append({"type": "table", "content": "\n".join(table_lines)})
            i = j
        else:
            text_buf.append(line)
            i += 1
    flush_text()
    return blocks
```

**核心逻辑:**一行以 `|` 开头 + 下一行也是 `|` → 连续的 `|` 行被识别为一个表格整体。表格作为一个完整 block 写入,**不进入后续的递归切分**。

**为什么不用 markdown 解析器?** md 解析器(如 `MarkdownHeaderTextSplitter`)会把表格的每一行当普通段落切,遇到 500 字限制就切断。用自己写的简单规则,适用范围更广(PDF/DOCX 转文本后的表格行也会以 `|` 开头)。

### 元数据继承机制

```python
base_meta = dict(doc.metadata)  # 父文档的 source/department/page
for block in blocks:
    if block["type"] == "table":
        doc_chunks.append(Document(page_content=block["content"],
                            metadata={**base_meta, "block_type": "table"}))
    else:
        sub = text_splitter.split_text(block["content"])
        for s in sub:
            doc_chunks.append(Document(page_content=s,
                                metadata={**base_meta, "block_type": "text"}))
```

**`{**base_meta, "block_type": "table"}`**:Python 的字典解包——把父文档的元数据原样继承到每个 chunk 上,再叠一个 `block_type` 标记。这样后面检索时,不管命中的是表格块还是文本块,都能拿到 source/department/page 等信息(权限过滤、引用溯源都需要)。

## 五、JWT 认证:`app/auth.py`

### 先看整体设计

```
客户端带 Authorization: Bearer <token> 请求
  → FastAPI 依赖注入触发 HTTPBearer → 取出 credentials
  → get_current_user 依赖 → verify_token
  → jwt.decode:验签名 + 验过期 + 取 sub
  → request.state.authenticated_user = user_id
  → 路由函数拿到 user_id 继续干活
```

**核心问题:为什么用 JWT,不用 Session?**

Session(服务端存状态)有三个硬伤:① 多实例部署时,请求落到 A 实例,A 的 session 里没有这次登录 → 要么上 Redis 共享 session,要么粘性路由,运维复杂度暴涨;② 水平扩容时每个实例都要复制 session 存储;③ 每次请求都要回查 session 存储,多一跳 IO。JWT 是**自包含**的——token 自己就带着"我是谁、什么时候过期",服务端只验签不查库,**无状态**天然适配水平扩容。

### payload 三个字段各管什么

```python
payload = {"sub": user_id, "iat": now, "exp": now + expires_hours * 3600}
```

| 字段 | 含义 | 作用 |
|---|---|---|
| `sub` | subject,用户标识(谁) | 整个鉴权的落脚点——验完签拿这个值当 user_id |
| `iat` | issued at,签发时刻 | 审计/排查"这个 token 什么时候发的";配合 exp 算有效时长 |
| `exp` | expiration,过期时刻 | `jwt.decode` 自动校验,过期抛 `ExpiredSignatureError` |

**为什么不用 `user_id` 当字段名?** JWT 是标准(RFC 7519),`sub`/`iat`/`exp` 是规范定义的保留 claim。用标准字段,任何符合规范的库/工具(jwt.io 调试、其他语言验签)都能直接读懂,不造私有轮子。

### 三重校验分别防什么攻击

```python
def verify_token(token, secret=""):
    secret = secret or DEFAULT_SECRET
    try:
        payload = jwt.decode(token, secret, algorithms=[JWT_ALGORITHM])  # ① 验签名
    except jwt.ExpiredSignatureError:                                     # ② 验过期
        raise HTTPException(401, "Token 已过期")
    except jwt.InvalidTokenError as e:                                    # ② 兜底
        raise HTTPException(401, f"Token 无效: {e}")
    user_id = payload.get("sub")                                          # ③ 验内容
    if not user_id:
        raise HTTPException(401, "Token 缺少用户标识")
    return user_id
```

| 校验层 | 防的攻击 | 不防会怎样 |
|---|---|---|
| ① 签名校验(`jwt.decode` 带 secret) | **伪造 token / 篡改 payload** | 攻击者改 `sub` 把自己升级成 admin;或凭空造一个合法 token |
| ② 过期校验(`exp`) | **token 泄露后被长期复用** | 抓包拿到 token,半年后还能用(盗号永久化) |
| ③ 内容校验(`sub` 非空) | **签了名但内容残缺的异常 token** | 业务拿到 `None` 的 user_id,下游 NPE |

**为什么是三重而不是一重?** 单验签(`jwt.decode`)看起来够,但:① 不带 `verify_exp` 的话过期 token 还能过;② 签名合法但 `sub` 为空的边界情况,decode 不报错却会让下游崩。**纵深防御**——签名管"真伪",过期管"时效",内容管"可用",每层兜不同的底。

### get_current_user:为什么用依赖注入拿 user_id

```python
async def get_current_user(request, credentials=security):
    secret = getattr(DEFAULT_CONFIG, "jwt_secret", "") or DEFAULT_SECRET
    if credentials is None or not credentials.credentials:
        raise HTTPException(401, "未提供认证 Token")
    user_id = verify_token(credentials.credentials, secret)
    request.state.authenticated_user = user_id  # 挂到 request 上,下游日志/审计能取
    return user_id
```

**关键:`user_id 是从 token 解码出来的,不是从请求体读的。** 如果从请求体/body 里读 `user_id`,那等于客户端自报家门——谁都能在 body 里填 `"user_id": "admin"`。依赖注入(`Depends(get_current_user)`)强制每个受保护路由**先过鉴权这一关**,拿到的是"被服务端验证过的、可信的" user_id。

**为什么不直接在每个路由里写 `verify_token(...)`?** 那样每个路由都要重复"取 header → 解析 → 验证 → 抛错"四步,漏一个路由就是一个未授权入口(坑)。依赖注入把这些收敛到一个函数,路由只声明 `user = Depends(get_current_user)`,**想保护谁就挂谁,不存在"忘了加校验"**。

### 为什么 `/health` 不加认证

```python
# 伪代码示意
@app.get("/health")
def health():           # 注意:没有 Depends(get_current_user)
    return {"status": "ok"}
```

**docker healthcheck / Kubernetes liveness/readiness 探针 / 监控系统(Prometheus)需要免认证访问。** 这些探针不带 token——它们是基础设施级的健康检查,不是业务请求。如果 `/health` 也要鉴权:① docker healthcheck 拿不到 token,容器一直被判 unhealthy → 被重启循环;② 监控系统要额外配 token 管理,复杂度上去了,而 `/health` 本身不泄露任何业务数据(就返回个 `ok`)。**公开但不敏感的端点,鉴权成本 > 收益。**

### `HTTPBearer(auto_error=False)` 的 `auto_error=False` 为什么重要

```python
security = HTTPBearer(auto_error=False)
```

`HTTPBearer` 默认 `auto_error=True`:没带 `Authorization` header 时,框架**直接抛 403**——你连写自定义错误信息的机会都没有,用户看到的是干巴巴的框架默认 403。

设成 `False` 后,没带 header 时 `credentials` 为 `None`,**控制权交回给我们**:

```python
if credentials is None or not credentials.credentials:
    raise HTTPException(401, "未提供认证 Token")  # 我们说了算:401 + 自定义文案
```

**401 vs 403 的区别在这里很关键:** "没带 token"(401 Unauthorized,可以补上 token 重试)和"带了但没权限"(403 Forbidden,带了对也没用)是两回事。`auto_error=False` 让我们能把"没带 token"正确归到 401,而不是被框架糊成 403。

## 六、LLM 降级链:`llm_fallback.py`

### 先看整体设计

```
FallbackLLM 包装多个底层 LLM(如 [primary(deepseek), backup(zai), emergency(gemini)])
  → invoke 时按 _ordered_llms() 排序逐个试
  → primary 挂了 → 标记冷却 60s → 自动切 backup
  → backup 成功 → 记住它是 _last_success_idx
  → 下次仍优先试 backup(不抖回 primary)
  → primary 冷却结束才会被排进候选
  → 全挂 → raise RuntimeError("所有 LLM 均不可用")
```

**核心问题:为什么用组合(`FallbackLLM` 持有一组 llm)不用继承(写个 `FallbackChatOpenAI`)?**

继承的硬伤:① 得继承一个具体的 LLM 类(`ChatOpenAI`/`ChatAnthropic`),但降级链里装的是**不同协议**的 LLM,没有共同父类可继承;② 继承后,LangChain 的 `Runnable` 接口、`usage_metadata`、`stream()` 等能力得自己重新对齐,容易漏。

组合的好处是 **duck-type 兼容**:`FallbackLLM` 只要内部那组 llm 都实现了 `.invoke()`/`.stream()`,对外就表现得像个 LLM。上层(Chain/Agent)调用它时,**完全不知道背后有降级**,照样能拿到 `usage_metadata`、能 `.stream()`。这就让降级链能**透明地插进任何用了 llm 的地方**,不破坏现有调用方代码。

### `_ordered_llms`:三段式排序逻辑

```python
def _ordered_llms(self):
    now = time.time()
    available, cooling = [], []
    idx = self._last_success_idx
    if self._cooldown[id(self._llms[idx])] <= now:           # ① 上次成功的,排第一
        available.append((idx, self._llms[idx]))
    for i, llm in enumerate(self._llms):
        if i == idx: continue
        if self._cooldown[id(llm)] <= now:                    # ② 其余未冷却的,按原序
            available.append((i, llm))
        else:                                                  # ③ 冷却中的,排最后
            cooling.append((i, llm))
    return available + cooling                                 # 可用的优先,冷却的垫底
```

排序结果永远是:`[上次成功且未冷却] + [其余未冷却] + [冷却中的]`。

**为什么"上次成功"要单独拎出来排第一?** 如果纯按原序(primary→backup→emergency),primary 短暂抖动恢复后,理论上该切回 primary——但你怎么知道它"真恢复了"而不是"这秒好了下秒又挂"?**与其赌,不如粘在最近成功的那台上**(sticky)。这就是下面的冷却 + 记忆机制要解决的。

### 熔断冷却:失败一次,冷却 60s

```python
except Exception as e:
    last_error = e
    self._cooldown[id(llm)] = time.time() + self._cooldown_seconds  # 60s 内不再试它
```

**不冷却会怎样?** primary 挂了,每来一个请求都先试 primary → 失败 → 才试 backup。等于**每个请求都白白付出一次"必定失败"的 primary 调用**(网络超时通常要好几秒),响应时间直接翻倍,还可能在 primary 雪崩时把它彻底压垮。

**冷却 60s 的效果:** primary 失败一次后,接下来 60s 内的请求 `_ordered_llms` 会把它排进 cooling 段(垫底),直接从 backup 起手。60s 后 primary 才重新进候选——给上游 API 留出恢复窗口。这是**客户端版的熔断器**(circuit breaker):不把故障的下游当死,但暂时不去碰它。

### `_last_success_idx`:为什么恢复后不立即切回 primary

```python
if idx != self._last_success_idx:
    self._last_success_idx = idx   # 换了成功的机器,记住它
```

假设 primary 抖动,切到 backup 成功了。如果逻辑是"primary 冷却结束就立刻切回 primary",会出现**抖动(thrashing)**:primary 好一下坏一下,流量在 primary↔backup 之间反复横跳,每次切换都是一次"赌注",还可能切回一个又坏的 primary。

**记忆 `_last_success_idx` 的好处:** 切到 backup 后就"粘"在 backup 上,即使 primary 冷却结束也**继续用 backup**,直到 backup 自己也挂了才会往下找。这叫 **sticky / 亲和性调度**——已经验证能用的就先用着,不为"理论上更优"的 primary 冒切换风险。代价是 primary 恢复后不会被"立即启用",换来的是**稳定的、可预测的服务路径**。

### stream 降级为什么比 invoke 更复杂

`invoke` 是一次性请求-响应:失败了换一个 provider 重试,无状态,干净利落。但 `stream` 是**流式输出**——已经吐了一半 token 给客户端了,这时 primary 挂了:

```
客户端已经收到: "根据公司规定,年假天数取决于..."
(primary 在这里挂了)
```

这时**不能"中途换 provider"**——backup 不会从"年假天数取决于..."接着往下续(它不知道前文,即使知道,token 级别的接续也几乎不可能对齐)。唯一办法是**整段重来**:从头用 backup 重新生成,但这要求客户端把已收到的部分丢弃,或者服务端先 buffer 完整响应再流出去(牺牲流式的实时性)。

**所以 stream 降级通常的策略是:** ① 流开始前的首 token 之前挂掉 → 可以换 provider 重试(还没向客户端吐东西);② 流中途挂掉 → 只能整段重来或报错让客户端重试,**没有优雅的"中途切换"**。这就是为什么很多系统对流式降级更保守(甚至直接不做降级,失败就失败)。

### 和 `llm_factory.py` 的关系:factory 造单个,fallback 包装多个

```
llm_factory.create_llm("deepseek")  → 1 个 ChatOpenAI 实例
llm_factory.create_llm("zai")       → 1 个 ChatAnthropic 实例
llm_factory.create_llm("gemini")    → 1 个 ChatGoogleGenerativeAI 实例

FallbackLLM([
    create_llm("deepseek"),   # primary
    create_llm("zai"),        # backup
    create_llm("gemini"),     # emergency
])
```

**职责分离:** `llm_factory.py` 负责"怎么造**一个** LLM 实例"(协议适配、参数注入、预设查找),`llm_fallback.py` 负责"怎么**编排多个**实例"(排序、冷却、记忆)。前者是**单点构造**,后者是**多点编排**。

这俩是**正交**的:factory 不关心你拿实例去单用还是包进降级链;fallback 不关心每个实例是 deepseek 还是 gemini,它只调 `.invoke()`。所以你可以自由组合——只造一个不降级也行,造三个叠成降级链也行,**互不耦合**。

## 总结:六个模块的设计脉络

| 模块 | 核心问题 | 解法 | 关键设计模式 |
|---|---|---|---|
| llm_factory | 怎么接入不同 LLM 又不改老代码 | 注册表+预设 | 工厂模式/装饰器注册 |
| retrieval | 怎么检索又准又安全 | 两阶段(粗排+精排)+权限前置过滤 | 管道模式(Pipeline) |
| cache | 怎么缓存快又不答错/不泄露 | 精确+语义+四重校验+权限签名 | 多层校验链 |
| data_prep | 怎么不让表格被切散 | 边界识别+整体保留+元数据继承 | 预切分 |
| auth | 怎么鉴权又无状态可扩容 | JWT 自包含 + 三重校验 + 依赖注入 | 依赖注入/纵深防御 |
| fallback | 怎么在 LLM 挂掉时还能答 | 多实例编排+冷却熔断+sticky 记忆 | 组合模式/熔断器 |
